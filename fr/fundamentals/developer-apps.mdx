<div id="overview">
  ## Aperçu
</div>

Si vous avez déjà des Apps, vous pouvez les consulter, les modifier ou les supprimer via la [page App du portail développeur](https://developer.x.com/en/portal/projects-and-apps).

L’accès à l’API X et à l’API X Ads nécessite un ensemble d’identifiants d’[authentification](/fr/resources/fundamentals/authentication), également appelés clés et jetons, que vous devez transmettre avec chaque requête. Ces identifiants peuvent prendre différentes formes selon le type d’authentification requis par l’endpoint spécifique que vous utilisez.

Voici les différents identifiants que vous pouvez générer dans votre App et comment les utiliser :

* **[API Key and Secret](/fr/resources/fundamentals/authentication#api-key-and-secret) :** En substance, le nom d’utilisateur et le mot de passe de votre App. Vous les utiliserez pour authentifier les requêtes qui nécessitent le [contexte utilisateur OAuth 1.0a](/fr/resources/fundamentals/authentication#oauth-1-0a-2), ou pour générer d’autres jetons tels que des Access Tokens utilisateur ou un App Access Token.

* **[Access Token and Secret](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) :** En général, les Access Tokens représentent l’utilisateur pour le compte duquel vous effectuez la requête. Ceux que vous pouvez générer via le portail développeur représentent l’utilisateur propriétaire de l’App. Vous les utiliserez pour authentifier les requêtes qui nécessitent le [contexte utilisateur OAuth 1.0a](/fr/resources/fundamentals/authentication#oauth-1-0a-2). Si vous souhaitez effectuer des requêtes pour le compte d’un autre utilisateur, vous devrez utiliser le flux OAuth à 3 volets afin qu’il vous autorise.

* **[Client ID and Client Secret](/fr/resources/fundamentals/authentication#oauth-2-0) :** Ces identifiants sont utilisés pour obtenir un Access Token utilisateur avec l’authentification OAuth 2.0. Comme avec OAuth 1.0a, les Access Tokens utilisateur servent à authentifier les requêtes qui fournissent des informations privées de compte utilisateur ou effectuent des actions pour le compte d’un autre compte, mais avec une portée fine pour un meilleur contrôle sur les accès de l’application cliente au compte utilisateur.

* **[App only Access Token](/fr/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) :** Vous utiliserez ce jeton pour les requêtes vers des endpoints qui renvoient des informations publiquement disponibles sur X.

En plus de générer les clés et jetons nécessaires pour effectuer des requêtes à l’API X, vous pourrez également définir des [autorisations](/fr/resources/fundamentals/developer-apps#app-permissions) d’accès, documenter le cas d’usage ou l’objectif de l’App, définir une [URL de rappel](/fr/resources/fundamentals/developer-apps#callback-urls) et modifier d’autres paramètres liés à votre environnement développeur de l’App depuis le [tableau de bord de gestion](https://developer.x.com/en/portal/projects-and-apps).

<div id="apps-and-projects">
  ### Applications et Projets
</div>

Vous pouvez utiliser des applications et des [projets](/fr/resources/fundamentals/projects) pour organiser votre travail avec la X Developer Platform par cas d’usage. Chaque projet peut inclure une application avec l’offre Free X API, jusqu’à deux applications avec l’offre Basic et jusqu’à trois applications avec l’offre Pro.

Si vous souhaitez accéder aux nouveaux endpoints de [X API v2](/fr/x-api/introduction), vous devrez utiliser des clés et des jetons issus d’une application associée à un projet.

Si vous avez des applications créées avant le lancement des projets, elles seront visibles dans la section intitulée « Standalone Apps ». Les Standalone Apps sont des applications en dehors de la structure des projets. Si vous rattachez une Standalone App à un projet, elle pourra ensuite effectuer des requêtes vers les endpoints v2.

<div id="developer-portal-dashboard">
  ### Tableau de bord du portail développeur
</div>

Vous pouvez [accéder au tableau de bord](https://developer.x.com/content/developer-twitter/en/apps) pour gérer les applications associées à votre compte. Pour en savoir plus, consultez notre page de documentation sur le [portail développeur](/fr/resources/fundamentals/developer-portal). Le tableau de bord permet aux développeurs d’effectuer rapidement et facilement les tâches suivantes :

* Afficher vos applications autonomes existantes et leur identifiant d’application (App ID) associé.
* Créer un nouveau projet, une application ou une application autonome.
* Supprimer un projet ou une application inutilisé(e).
* Examiner ou mettre à jour les paramètres d’une application spécifique, notamment le nom, la description, le site web, l’URL de rappel et les [autorisations](/fr/resources/fundamentals/developer-apps#app-permissions).
* Régénérer des identifiants spécifiques à l’application, tels que la clé et le secret d’API, le jeton d’accès de l’application, ainsi que les jetons d’accès utilisateur du propriétaire de l’application.

<div id="signing-up-for-access">
  ### Inscription et accès
</div>

Si vous avez déjà des X Apps, vous pouvez consulter et modifier vos Apps via le [tableau de bord des Apps](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) de X si vous êtes connecté à votre compte X sur developer.x.com. Notez que vous **n’avez pas** besoin de créer un compte pour gérer toutes les Apps précédemment créées sur developer.x.com.

Si vous devez créer une nouvelle X App, vous devez d’abord [ouvrir un compte développeur](https://developer.x.com/en/portal/petition/essential/basic-info). Si vous êtes membre d’un [compte d’équipe](/fr/resources/fundamentals/developer-portal#team-management), un rôle d’administrateur doit vous être attribué pour pouvoir créer une nouvelle App.

<div id="automated-account-labeling-for-bots">
  ### Libellé « Compte automatisé » pour les bots
</div>

Vous pouvez ajouter un libellé « Compte automatisé » à vos comptes bot pour indiquer aux utilisateurs sur X que votre bot est un compte automatisé. Ces bots exécutent des actions programmées via l’API X. En ajoutant ce libellé à votre bot, vous renforcez la confiance de votre audience, légitimez votre compte et vous distinguez des bots indésirables. Cela aide les personnes sur X à mieux comprendre la finalité de votre compte lorsqu’elles interagissent avec votre bot.

Pour ajouter un libellé à votre compte bot, suivez ces étapes :

1. Accédez aux paramètres de votre compte
2. Sélectionnez « Votre compte »
3. Sélectionnez « Automatisation »
4. Sélectionnez « Gestion du compte »
5. Sélectionnez ensuite le compte X qui exécute votre bot
6. Saisissez votre mot de passe pour vous connecter
7. Enfin, une confirmation devrait indiquer que le libellé a été appliqué à votre compte.

  ## Gestion des applications
</div>

<div id="introduction">
  ### Introduction
</div>

Le [tableau de bord X App](https://developer.x.com/en/portal/projects-and-apps) permet aux développeurs d’effectuer rapidement et facilement les tâches suivantes :

* Afficher vos applications et [projets](/fr/resources/fundamentals/projects) existants ainsi que leur identifiant d’application (App ID) associé.
* Créer un nouveau projet ou une application autonome.
* Supprimer un projet, une application ou une application autonome.
* Ouvrir les paramètres d’une application spécifique en accédant aux paramètres de l’application. Dans les paramètres, vous pouvez voir les détails de l’application, les clés et jetons, ainsi que les autorisations.
* Mettre à jour les paramètres d’authentification utilisateur de votre application pour utiliser OAuth 1.0a ou OAuth 2.0.

<Note>
  **Remarque :**

  Toutes les clés et tous les jetons d’application ne sont plus visibles dans le Developer Portal et doivent être enregistrés en lieu sûr dès leur génération. Nous recommandons d’utiliser un gestionnaire de mots de passe pour stocker vos clés et jetons.

  Vous pouvez afficher un indice de clé API pour vous aider à faire correspondre vos identifiants à l’application correspondante.
</Note>

<div id="app-settings">
  ### Paramètres de l’application
</div>

<div id="app-details">
  #### Détails de l'application
</div>

Ici, vous pouvez modifier l’icône de l’application, le nom de l’application, la description de l’application, l’URL de votre site web, les [URLs de rappel/URIs de redirection](/fr/resources/fundamentals/developer-apps#callback-urls), l’URL des conditions d’utilisation, l’URL de la politique de confidentialité, le nom de l’organisation, l’URL de l’organisation, ainsi que l’objectif ou le cas d’utilisation de l’application.

OAuth 2.0 et OAuth 1.0a sont des méthodes d’authentification qui permettent aux utilisateurs de se connecter à votre application avec X. Elles permettent également à votre application d’effectuer des requêtes spécifiques au nom d’utilisateurs authentifiés. Vous pouvez activer l’une ou l’autre, ou les deux méthodes pour votre application.

Il est important de maintenir ces informations à jour. Le nom de votre application et l’URL de votre site web seront affichés comme source dans les métadonnées de tout Tweet créé par programmation par votre application. Si vous modifiez le cas d’utilisation d’une application X, veillez à mettre à jour ce cas d’utilisation dans ces paramètres afin de garantir votre conformité aux [Conditions du développeur](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy).

Si votre application affiche la mention « suspended », c’est parce qu’elle enfreint une ou plusieurs des [conditions pour les développeurs](https://developer.x.com/en/developer-terms.html) de X, telles que nos [règles d’automatisation](https://help.x.com/en/rules-and-policies/twitter-automation). L’équipe des politiques de la plateforme développeur communiquera avec les développeurs via l’adresse e-mail configurée sur le compte X du propriétaire de l’application ; pour vérifier cette adresse e-mail, veuillez consulter vos [paramètres de compte X](https://x.com/settings/your_twitter_data/account). Les e-mails de notification concernant les suspensions seront envoyés à cette adresse avec un objet similaire à « Application suspension notice » et contiendront des informations spécifiques sur la marche à suivre. Pour travailler avec l’équipe X afin de résoudre les suspensions, veuillez vérifier votre e-mail pour des instructions spécifiques, ou utilisez notre [formulaire d’aide de la plateforme](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Clés et jetons
</div>

Dans une application au sein d’un projet ou via une application autonome, vous pouvez afficher, régénérer ou révoquer les jetons suivants :

* [API Key (Consumer Key) et API Secret (Consumer Secret)](/fr/resources/fundamentals/authentication#api-key-and-secret)
* [App Access Token](/fr/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
* [User Access Token and Access Token Secret](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  L’Access Token et le Secret disponibles dans le portail développeur se rapportent à l’utilisateur propriétaire de l’application.

**Veuillez noter** - Si vous souhaitez effectuer des requêtes au nom d’un autre utilisateur (autrement dit, un utilisateur autre que le propriétaire de l’application), vous devrez utiliser soit le [flux OAuth 1.0a en 3 étapes](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens), soit le flux OAuth 2.0 [Authorization Code avec PKCE](/fr/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) pour générer un ensemble de jetons d’accès utilisateur. Vous utiliserez ensuite ces jetons spécifiques à l’utilisateur dans votre requête à l’API.

<div id="user-authentication-settings">
  #### Paramètres d’authentification utilisateur
</div>

Vous pouvez configurer l’authentification de votre application avec OAuth 1.0a ou OAuth 2.0. OAuth 2.0 est compatible uniquement avec l’API X v2. OAuth 2.0 vous permet de sélectionner des autorisations fines (scopes) qui accordent des permissions spécifiques au nom d’un utilisateur. OAuth 1.0a peut être utilisé avec les API X v1.1 et v2 et repose sur une autorisation plus large avec des scopes plus généraux.

<div id="oauth-10a-application-user-permissions">
  #### Autorisations OAuth 1.0a application–utilisateur
</div>

Si vous êtes le propriétaire de l’app, vous pouvez ajuster ses autorisations (lecture seule ; lecture et écriture ; ou lecture, écriture et messages privés). Cela détermine les ressources et les événements auxquels vous avez accès via les API X (remarque : certaines ressources nécessitent une autorisation supplémentaire accordée directement par X).

Vous pouvez également activer ou désactiver, sur cette page, la capacité de vos apps à demander les adresses e‑mail des utilisateurs (cela nécessite qu’une URL des Conditions d’utilisation et une URL de la Politique de confidentialité soient renseignées sur la page « App details »).

<div id="oauth-20-type-of-app">
  #### Type d’application OAuth 2.0
</div>

Si vous choisissez OAuth 2.0 comme méthode d’authentification des utilisateurs, vous devrez sélectionner le type d’application que vous créez. Vos options sont Application native, Application monopage, Application web, Application automatisée ou bot. Les applications natives et les applications monopage sont des clients publics, tandis que l’application web et l’application automatisée (ou les bots) sont des clients confidentiels.

Les clients confidentiels s’authentifient de manière sécurisée auprès du serveur d’autorisation. Ils protègent votre secret client. Les clients publics sont des applications généralement exécutées dans un navigateur ou sur un appareil mobile et ne peuvent pas utiliser vos secrets client. Si vous sélectionnez un type d’application qui est un client confidentiel, un secret client vous sera fourni.

<div id="app-permissions">
  ## Autorisations de l’application
</div>

<div id="oauth-10a-app-permissions">
  ### Autorisations d’application OAuth 1.0a
</div>

Les autorisations d’application décrivent le niveau d’accès pour l’authentification application‑utilisateur OAuth 1.0a. Elles sont configurées par application dans les paramètres de votre [X App](/fr/resources/fundamentals/developer-apps).

Trois niveaux d’autorisation sont disponibles :

1. Lecture seule
2. Lecture et écriture
3. Lecture, écriture et accès aux Messages privés

Une autorisation supplémentaire permet de demander l’accès à l’adresse e‑mail d’un utilisateur — elle peut être combinée avec n’importe lequel des trois niveaux ci‑dessus.

Si un niveau d’autorisation est modifié, tous les jetons utilisateur déjà émis pour cette application X doivent être révoqués et les utilisateurs doivent à nouveau autoriser l’application afin que le jeton hérite des autorisations mises à jour.

Il est recommandé de ne demander que le niveau d’accès minimal aux données du compte d’un utilisateur nécessaire à une application ou un service.

<div id="read-only">
  ### Lecture seule
</div>

Ce niveau d’autorisation permet un accès en lecture aux ressources X, notamment (par exemple) aux Tweets d’un utilisateur, à son fil d’accueil et à ses informations de profil. Il n’autorise pas la lecture des Messages privés d’un utilisateur et ne permet la mise à jour d’aucun élément ou objet.

<div id="read-and-write">
  ### Lecture et écriture
</div>

Ce niveau d’autorisation permet un accès en lecture et en écriture aux ressources X. En plus de l’accès en lecture, il permet de publier des posts, de suivre des comptes et de mettre à jour des éléments du profil d’un utilisateur. Il permet également de masquer des réponses au nom de l’utilisateur authentifié. Ce niveau d’autorisation n’autorise aucun accès aux Messages privés (y compris la lecture, l’écriture ou la suppression).

<div id="read-write-and-access-direct-messages">
  ### Lire, écrire et accéder aux messages privés
</div>

Ce niveau d’autorisation inclut tout ce qui précède et ajoute la possibilité de lire, d’écrire et de supprimer des messages privés au nom d’un utilisateur.

* [POST /2/dm_conversations/:dm_conversation_id/messages](/fr/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
* [POST /2/dm_conversations/](/fr/x-api/direct-messages/create-a-new-dm-conversation)
* [POST /2/dm_conversations/with/:participant_id/messages](/fr/x-api/direct-messages/send-a-new-message-to-a-user)
* [GET /2/dm_conversations/with/:user_id/dm_events](/fr/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
* [GET /2/dm_conversations/:dm_conversation_id/dm_events](/fr/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
* [GET /2/dm_events](/fr/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Détermination des autorisations
</div>

Toutes les requêtes API authentifiées renvoient un en-tête `x-access-level` dans la réponse HTTP. La valeur de cet en-tête indique le niveau d’autorisation actuellement utilisé. Les valeurs possibles sont read, read-write et read-write-directmessages.

<div id="callback-urls">
  ## URL de rappel
</div>

Les méthodes d’authentification OAuth 1.0a User Context et OAuth 2.0 Authorization Code avec PKCE permettent aux développeurs d’effectuer des requêtes au nom de différents utilisateurs X ayant suivi un parcours de connexion spécifique. Actuellement, deux parcours peuvent être utilisés pour permettre aux utilisateurs d’autoriser votre application :

* [Parcours de code d’autorisation OAuth 2.0 avec PKCE](/fr/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
* [Parcours OAuth 1.0a « 3-legged OAuth »](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (et séparément, le [parcours « Se connecter avec X »](/fr/resources/fundamentals/authentication#log-in-with-x))

Au fur et à mesure que les utilisateurs suivent ces parcours, ils doivent être redirigés vers une page web ou un emplacement après s’être connectés avec succès et avoir autorisé l’App du développeur. Cette page ou cet emplacement de redirection est appelé URL de rappel.

Lors de la configuration de ces parcours pour leurs utilisateurs potentiels, les développeurs doivent transmettre une URL de rappel avec leurs requêtes aux points de terminaison d’authentification qui composent les parcours mentionnés précédemment. Par exemple, les développeurs utilisant OAuth 1.0a User Context doivent fournir le paramètre `callback_url` lors d’une requête vers le point de terminaison [GET oauth/request_token](/fr/resources/fundamentals/authentication#post-oauth-request-token). De même, les développeurs utilisant OAuth 2.0 Authorization Code avec PKCE doivent fournir le paramètre `redirect_uri` avec leur requête vers le point de terminaison GET oauth2/authorize.

En plus d’utiliser ces paramètres, le développeur doit également s’assurer que l’URL de rappel a été ajoutée à la liste d’autorisation des URL de rappel de leur [App](/fr/resources/fundamentals/developer-apps), accessible depuis la page des paramètres de l’App sur le [developer portal](/fr/resources/fundamentals/developer-portal).

Si la configuration est correcte, les développeurs seront redirigés vers l’URL de rappel après s’être connectés avec succès à X dans le cadre de ces parcours.

<div id="things-to-keep-in-mind">
  ### Points à garder à l’esprit
</div>

Lorsque vous configurez vos URL de rappel (callback), voici quelques points à garder à l’esprit :

**Encodez vos paramètres de requête en HTTP**

Comme vous transmettez l’URL de rappel en tant que paramètre de requête via `callback_url` ou `redirect_uri`, veillez à encoder l’URL en HTTP.

**Limites des URL de rappel**

Il existe une limite stricte de 10 URL de rappel dans le tableau de bord X Apps. Votre URL de rappel doit toujours correspondre exactement entre l’URL autorisée que vous ajoutez dans le tableau de bord Apps et le paramètre que vous utilisez dans le flux d’autorisation.

Si vous souhaitez inclure des données propres à la requête dans l’URL de rappel, vous pouvez utiliser le paramètre `state` pour stocker des données qui seront incluses après la redirection de l’utilisateur. Vous pouvez soit encoder les données directement dans le paramètre `state`, soit utiliser ce paramètre comme identifiant de session pour stocker l’état côté serveur.

**N’utilisez pas localhost comme URL de rappel**
Au lieu d’utiliser localhost, utilisez un hôte personnalisé en local ou http(s)://127.0.0.1

**URL à protocole personnalisé**
Si vous souhaitez tirer parti du deep linking mobile, vous pouvez utiliser des URL à protocole personnalisé avec une partie chemin et domaine, comme twitter://callback/path. Toutefois, certaines valeurs de protocole sont interdites et doivent être évitées. Vous trouverez la liste des protocoles interdits ci-dessous.

<div id="error-example">
  ### Exemple d’erreur
</div>

Si vous utilisez une URL de redirection qui n’a pas été correctement ajoutée aux paramètres de votre application dans le portail développeur, vous recevrez le message d’erreur suivant :

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

OU

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**Dépannage**

Si vous rencontrez une erreur, vérifiez que l’URL de redirection utilisée dans vos requêtes d’authentification est encodée correctement (URL-encoded) et qu’elle correspond exactement à une URL de redirection ajoutée à la liste d’autorisation (allowlist) de l’application dont vous utilisez les clés et jetons dans votre requête.
