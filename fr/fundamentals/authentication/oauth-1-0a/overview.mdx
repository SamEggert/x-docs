<div id="oauth-10a">
  ## OAuth 1.0a
</div>

De nombreux points de terminaison sur la plateforme des développeurs X utilisent la méthode OAuth 1.0a pour agir, ou effectuer des requêtes API, au nom d’un compte X. Par exemple, si vous avez une application développeur X, vous pouvez effectuer des requêtes API au nom de n’importe quel compte X, à condition que cet utilisateur authentifie votre application.

Veuillez noter : si vous n’êtes pas à l’aise avec des notions telles que HMAC-SHA1 et l’encodage pourcentage, nous vous recommandons de consulter la section « outils utiles » ci-dessous, qui répertorie certains clients API simplifiant grandement le processus d’authentification.

<div id="key-concepts">
  ### Concepts fondamentaux
</div>

<div id="signing-a-request-with-keys-and-tokens">
  #### Signature d’une requête avec des clés et des jetons
</div>

Vous devez signer chaque requête API en transmettant plusieurs clés et jetons générés dans un en-tête d’autorisation. Pour commencer, vous pouvez générer plusieurs clés et jetons sur la page de détails de votre [application développeur X](/fr/resources/fundamentals/developer-apps), notamment :

|     |     |
| :--- | :--- |
| Clé API et secret :<br /><br />`oauth_consumer_key`<br /><br />`oauth_consumer_secret` | Considérez-les comme le nom d’utilisateur et le mot de passe qui identifient votre application développeur X lors des requêtes API. |
| Jeton d’accès et secret :<br /><br />`oauth_token`<br /><br />`oauth_token_secret` | Un jeton d’accès et un secret de jeton d’accès sont des informations d’identification propres à un utilisateur, utilisées pour authentifier les requêtes API OAuth 1.0a. Ils indiquent le compte X pour le compte duquel la requête est effectuée.<br /><br />Vous pouvez générer votre propre jeton d’accès et secret de jeton si vous souhaitez que votre application effectue des requêtes pour le compte du même compte X associé à votre compte développeur sur la page de détails de l’[application développeur X](/fr/resources/fundamentals/developer-apps).<br /><br />Si vous souhaitez générer des jetons d’accès pour un autre utilisateur, consultez « Effectuer des requêtes pour le compte d’utilisateurs » ci-dessous. |

<div id="making-requests-on-behalf-of-users">
  #### Effectuer des requêtes pour le compte d’utilisateurs
</div>

Pour créer une signature, vous avez besoin d’un ensemble de jetons d’accès représentant l’utilisateur pour le compte duquel vous allez effectuer une requête.

Vous pouvez générer un ensemble de jetons d’accès représentant le compte X qui possède l’application X Developer depuis la [page des détails de l’appli](https://developer.x.com/content/developer-twitter/en/apps). En revanche, si vous souhaitez effectuer une requête pour le compte d’un autre compte X, le propriétaire de ce compte doit vous accorder l’accès en se connectant à son compte dans le cadre du [flux OAuth à 3 étapes](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Le résultat de ce processus est un ensemble de jetons d’accès (oauth_token et oauth_token_secret) pouvant être utilisés pour effectuer une requête OAuth 1.0a.

Une fois ces clés et jetons en main, vous pouvez soit [créer une signature](/fr/resources/fundamentals/authentication#creating-a-signature) à partir de zéro (ce que nous ne recommandons pas, sauf si vous savez ce que vous faites), soit utiliser l’un des outils mentionnés ci-dessous pour effectuer une requête vers un endpoint nécessitant OAuth 1.0a.

À titre de référence, voici un exemple de requête cURL signée, avec tous les jetons générés passés dans un en-tête d’autorisation :

```bash
curl --request POST \
  --url 'https://api.x.com/1.1/statuses/update.json?status=Hello%20world' \
  --header 'authorization: OAuth oauth_consumer_key="CONSUMER_API_KEY", oauth_nonce="OAUTH_NONCE", oauth_signature="OAUTH_SIGNATURE", oauth_signature_method="HMAC-SHA1", oauth_timestamp="OAUTH_TIMESTAMP", oauth_token="ACCESS_TOKEN", oauth_version="1.0"' \
```

<Note>
  **Remarque :** Les jetons d’accès utilisateur sont sensibles et doivent être protégés avec le plus grand soin. Lorsque des jetons d’accès sont générés, l’utilisateur qu’ils représentent fait confiance à votre application pour les sécuriser. Si la sécurité des clés API et des jetons d’accès utilisateur est compromise, votre application pourrait potentiellement exposer des informations privées et des fonctionnalités de compte. Nous vous encourageons à en apprendre davantage sur la [sécurisation des clés et des jetons d’accès](/fr/resources/fundamentals/developer-apps).
</Note>

<div id="useful-tools">
  ### Outils utiles
</div>

Le processus de [signature d’une requête](/fr/resources/fundamentals/authentication#creating-a-signature) est complexe. Nous vous recommandons d’utiliser une bibliothèque cliente pour API qui génère automatiquement une grande partie des informations d’authentification :

|     |     |
| :--- | :--- |
| [Postman](https://www.getpostman.com/) | Un client API qui vous permet de construire et d’envoyer des requêtes REST. Lisez notre tutoriel « [Premiers pas avec Postman](/fr/tutorials/postman-getting-started) » pour en savoir plus sur cet outil. |
| [Insomnia](https://insomnia.rest/) | Insomnia est un client REST avec gestion des cookies, variables d’environnement, génération de code et authentification pour Mac, Windows et Linux. |
