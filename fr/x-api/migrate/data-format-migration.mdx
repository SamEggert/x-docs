<div id="introduction">
  ## Introduction
</div>

Avec le lancement de la v2 de l’API X, nous avons adopté un nouveau format de réponse et une nouvelle méthode pour demander différents objets et champs, que nous appelons simplement le format X API v2.

Dans la section sur les différences générales, vous pouvez découvrir certains changements pertinents pour les utilisateurs standard et enterprise. Nous avons également préparé un guide spécifique pour le [format natif v1.1 (standard)](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), le [format Native Enriched (enterprise)](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) et le [format Activity Streams (enterprise)](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object), qui aide à faire correspondre les champs et explique quels champs et expansions utiliser pour demander les nouveaux champs v2.

* [Format natif vers X API v2 (v1.1 standard)](/fr/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)
* [Native Enriched vers X API v2 (enterprise)](/fr/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
* [Activity Streams vers X API v2 (enterprise)](/fr/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

Vous pouvez également être intéressé par notre [outil visuel de migration des formats de données](/fr/x-api/migrate/data-format-migration#visual-data-format-migration-tool), qui vous aide à voir rapidement les différences entre le [format de données X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) et le [format X API v2](/fr/x-api/introduction).

<div id="general-differences">
  ### Différences générales
</div>

<div id="requesting-objects-and-fields">
  #### Requête d’objets et de champs
</div>

L’une des principales différences entre les endpoints pré‑v2 et v2 est que la nouvelle version ne renvoie que quelques champs par défaut, tandis que les endpoints standard, premium et enterprise renvoient la plupart des champs par défaut. La nouvelle version utilise des paramètres appelés [fields](/fr/x-api/fundamentals/fields) et [expansions](/fr/x-api/fundamentals/expansions) pour demander explicitement des données supplémentaires au-delà des valeurs par défaut, ce qui vous permet de ne récupérer que les données dont vous avez besoin sans devoir ingérer des champs qui ne vous concernent pas.

Tous les champs que vous demandez et qui se rapportent à l’objet de données principal seront renvoyés dans cet objet principal, avec les valeurs par défaut. En revanche, si vous demandez des objets étendus à l’aide du paramètre expansions, les objets secondaires seront renvoyés dans un nouvel objet includes. Vous pouvez faire correspondre les objets étendus présents dans l’objet includes à l’objet principal en utilisant le champ id, qui sera renvoyé dans les deux.

Par exemple, si vous utilisez l’endpoint v2 [Post lookup](/fr/x-api/posts/lookup/introduction) et que vous incluez le paramètre expansions=author&#95;id dans votre requête, vous recevrez le champ author&#95;id dans l’objet Post principal, ainsi qu’un objet user par Post dans l’objet includes, chacun incluant le champ id par défaut qui peut être utilisé pour faire correspondre l’objet user à l’objet Post. Voici un exemple de ce à quoi cela ressemble :

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### Modèle JSON mis à jour
</div>

En plus des changements concernant la façon de demander certains champs, X API v2 introduit également de nouveaux modèles JSON pour les objets renvoyés par les API, notamment les objets [Post](/fr/x-api/fundamentals/data-dictionary#tweet) et [user](/fr/x-api/fundamentals/data-dictionary#user).

* Au niveau racine du JSON, les endpoints standard renvoient des objets Post dans un tableau **statuses**, tandis que X API v2 renvoie un tableau **data**.
* Au lieu de faire référence à des « statuses » Retweeted et Quoted, le JSON de X API v2 fait référence à des Tweets retweetés et cités. De nombreux champs hérités et obsolètes, tels que **contributors** et **user.translator&#95;type**, sont supprimés.
* Au lieu d’utiliser à la fois **favorites** (dans l’objet Post) et **favourites** (dans l’objet user), X API v2 utilise le terme like.
* X adopte la convention selon laquelle les valeurs JSON sans valeur (par exemple null) ne sont pas écrites dans la charge utile. Les attributs Post et user ne sont inclus que s’ils ont des valeurs non nulles. 
   

<div id="new-v2-fields">
  #### Nouveaux champs v2
</div>

Nous avons également introduit un nouvel ensemble de champs dans l’[objet Post](/fr/x-api/fundamentals/data-dictionary#tweet), notamment :

* Un champ [conversation&#95;id](/fr/x-api/fundamentals/conversation-id)
* Deux nouveaux champs [annotations](/fr/x-api/fundamentals/post-annotations), à savoir context et entities
* Plusieurs nouveaux champs [metrics](/fr/x-api/fundamentals/metrics)
* Un nouveau champ reply&#95;setting, qui indique qui est autorisé à répondre à une publication donnée

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### Migration du format de données standard v1.1 vers v2
</div>

Si ce n’est pas déjà fait, nous vous recommandons de commencer par lire l’introduction à la [migration des formats de données](/fr/x-api/migrate/data-format-migration). Vous pouvez également consulter notre [outil visuel de migration des formats de données](/fr/x-api/migrate/data-format-migration#visual-data-format-migration-tool) pour comparer rapidement le [format de données de l’API X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) et le [format de l’API X v2](/fr/x-api/fundamentals/data-dictionary).

Le format de données standard v1.1, également appelé format natif, est le principal format diffusé via les endpoints [standard v1.1](https://developer.x.com/en/docs/twitter-api/v1).

Si vous utilisez l’offre Premium, veuillez consulter le [guide du format natif enrichi](/fr/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2). Les clients Enterprise peuvent utiliser le format natif enrichi ou les flux d’activités, selon la configuration de votre espace dans la console Gnip.

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### Migration du format de données Native Enriched vers v2
</div>

Le format de données Native Enriched est utilisé par nos produits [Enterprise](/fr/x-api/enterprise-gnip-2.0/enterprise-gnip).

Le format de données Native Enriched a été mis à jour pour fournir des métadonnées de Tweets édités. Pour en savoir plus sur les métadonnées d’édition de Tweets, consultez la page [Principes de base de l’édition de Tweets](/fr/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si vous utilisez les endpoints standard v1.1, veuillez consulter le [guide de migration de la version standard v1.1 vers v2](/fr/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2). Si vous utilisez les produits Enterprise avec Activity Streams, nous proposons également un guide [Activity Streams vers v2](/fr/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2).

X API v2 introduit de nouveaux schémas JSON pour les objets [Tweet](/fr/x-api/fundamentals/data-dictionary#tweet) et [user](/fr/x-api/fundamentals/data-dictionary#user).

* Au niveau racine du JSON, le format Native Enriched renvoie des objets Tweet dans un tableau results, tandis que X API v2 renvoie un tableau data.
* Au lieu d’utiliser à la fois favorites (dans l’objet Tweet) et favourites (dans l’objet user), X API v2 utilise le terme like.
* X adopte la convention selon laquelle les valeurs JSON sans valeur (par exemple null) ne sont pas incluses dans la charge utile. Les attributs des objets Tweet et user ne sont présents que s’ils ont des valeurs non nulles.
* Tous les champs id en v2 seront des chaînes de caractères

En plus des modifications apportées au nouveau format JSON, nous avons également introduit un nouvel ensemble de champs dans l’objet Tweet, notamment :

* conversation&#95;id
* reply&#95;settings
* alt&#95;text sur les médias
* Deux nouveaux champs [annotations](/fr/x-api/fundamentals/post-annotations), à savoir context et entities
* Plusieurs nouveaux champs [metrics](/fr/x-api/fundamentals/metrics)
* Plusieurs nouveaux champs [polls](/fr/x-api/fundamentals/data-dictionary#poll)

De nombreux champs hérités et obsolètes sont supprimés :

* contributors
* Certains champs entities.media et extended&#95;entities.media
* filter&#95;level
* timestamp&#95;ms
* truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Migration du format de données Activity Streams vers v2
</div>

Le format de données Activity Streams est disponible avec nos produits [Enterprise](/fr/x-api/enterprise-gnip-2.0/enterprise-gnip).

Le format de données Activity Streams a été mis à jour afin d’inclure les métadonnées des Tweets modifiés. Pour en savoir plus sur les métadonnées de modification de Tweet, consultez la page [Principes de base des Tweets modifiés](/fr/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si vous utilisez les endpoints standard v1.1, veuillez consulter le [guide de migration de la version standard v1.1 vers v2](/fr/x-api/migrate/overview). Si vous utilisez les endpoints Premium, ou le format Native Enriched pour Enterprise, veuillez consulter le [guide de migration de Native Enriched vers v2](/fr/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2).

X API v2 introduit de nouveaux schémas JSON pour les objets [post](/fr/x-api/fundamentals/data-dictionary#tweet) et [user](/fr/x-api/fundamentals/data-dictionary#user).

* Au niveau racine du JSON, le format Activity Streams renvoie des objets Tweet dans un tableau results, tandis que X API v2 renvoie un tableau data.
* Au lieu de faire référence aux « activités » Retweeted et Quoted, le JSON de X API v2 fait référence aux Tweets Retweeted et Quoted.
* Au lieu d’utiliser à la fois favorites (dans l’objet Tweet) et favourites (dans l’objet user), X API v2 utilise le terme like.
* X adopte la convention selon laquelle les valeurs JSON sans valeur (par exemple null) ne sont pas incluses dans la charge utile. Les attributs de Tweet et de user ne sont inclus que s’ils ont des valeurs non nulles.
* Tous les champs id dans v2 seront de type chaîne.
   

En plus des changements apportés au nouveau format JSON, nous avons également introduit un nouvel ensemble de champs dans l’objet Tweet, notamment :

* conversation&#95;id
* reply&#95;settings
* alt&#95;text sur les médias
* Deux nouveaux champs [annotations](/fr/x-api/fundamentals/post-annotations), à savoir context et entities
* Plusieurs nouveaux champs [metrics](/fr/x-api/fundamentals/metrics)
* Plusieurs nouveaux champs [polls](/fr/x-api/fundamentals/data-dictionary#poll)
   

De nombreux champs hérités et obsolètes sont supprimés ou remplacés :

* display&#95;text&#95;range
* generator
* gnip
* link
* objectType
* provider
* twitter&#95;entities.symbols remplacé par data.entities.cashtags
* Certains champs twitter&#95;extended&#95;entities.media et twitter&#95;entities.media
* twitter&#95;filter&#95;level
* twitterTimeZone
* verb
