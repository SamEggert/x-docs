<div id="tweet">
  ### Tweet
</div>

Les Tweets constituent l’élément de base de tout ce qui concerne Twitter. L’objet Tweet possède une longue liste de champs « de premier niveau », tels que `id`, `text` et `created_at`. Les objets Tweet sont également l’objet « parent » de plusieurs objets enfants, notamment `user`, `media`, `poll` et `place`. Utilisez le paramètre de champ `tweet.fields` lorsque vous demandez ces champs de premier niveau sur l’objet Tweet.

L’objet Tweet peut être recherché et étendu dans la ressource utilisateur. Des Tweets supplémentaires liés au Tweet demandé peuvent aussi être trouvés et étendus dans la ressource Tweet. L’objet est disponible via l’extension `?expansions=pinned_tweet_id` dans la ressource utilisateur ou `?expansions=referenced_tweets.id` dans la ressource Tweet pour obtenir l’objet avec uniquement les champs par défaut. Utilisez cette extension avec le paramètre de champ `tweet.fields` lorsque vous demandez des champs supplémentaires afin de compléter l’objet.

| Field Value           | Type   | Description                                                                                                                           | How it Can Be Used                                                  |
|:-----------------------|:--------|:---------------------------------------------------------------------------------------------------------------------------------------|:----------------------------------------------------------------------|
| **id (default)**      | string | L’identifiant unique du Tweet demandé.                                                                                               | Utilisez ceci pour récupérer par programmation un Tweet précis.      |
| **text (default)**    | string | Le texte UTF-8 du Tweet. Voir [twitter-text](https://github.com/twitter/twitter-text/) pour les caractères autorisés.               | Extraction de mots-clés et analyse/classification des sentiments.    |
| **edit_history_tweet_ids (default)** | object | Identifiants uniques indiquant toutes les versions d’un Tweet. Pour les Tweets sans modification, il n’y aura qu’un ID. Pour les Tweets avec un historique de modifications, il y aura plusieurs ID. | Utilisez ces informations pour retrouver l’historique des modifications d’un Tweet. |
| **attachments**       | object | Indique le type de pièces jointes (le cas échéant) présentes dans ce Tweet.                                                         | Comprendre les objets renvoyés par les expansions demandées.         |
| **author_id**         | string | L’identifiant unique de l’utilisateur ayant publié ce Tweet.                                                                         | Hydrater l’objet User, partager un jeu de données pour relecture par les pairs. |
| **context_annotations** | array | Contient les annotations de contexte du Tweet.                                                                                      | Reconnaissance/extraction d’entités, analyse thématique.             |
| **conversation_id**   | string | L’ID du Tweet d’origine de la conversation (qui inclut les réponses directes et les réponses aux réponses).                         | Utilisez ceci pour reconstruire la conversation à partir d’un Tweet. |
| **created_at**        | date (ISO 8601) | Date et heure de création du Tweet.                                                                                               | Utile pour l’analyse de séries temporelles et pour savoir quand un Tweet a été créé. |
| **edit_controls**     | object | Indique pendant combien de temps le Tweet peut encore être modifié et le nombre de modifications restantes.                         | Utilisez ceci pour déterminer si un Tweet est éligible à la modification. |
| **entities**          | object | Entités extraites du texte du Tweet. Voir entities dans Twitter Objects.                                                             | Fournit des informations supplémentaires sur les hashtags, URL, mentions, etc. |
| **in_reply_to_user_id** | string | Si le Tweet représenté est une réponse, ce champ contient l’ID de l’auteur du Tweet d’origine.                                     | Déterminer si un Tweet était une réponse à un autre Tweet.           |
| **lang**              | string | Langue du Tweet, si elle est détectée par Twitter.                                                                                   | Classer les Tweets par langue parlée.                                |
| **non_public_metrics** | object | Mesures d’engagement non publiques pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur.  | Déterminer le nombre total d’impressions générées pour le Tweet.     |
| **organic_metrics**   | object | Mesures d’engagement suivies dans un contexte organique pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur. | Mesurer l’engagement organique du Tweet.                             |
| **possibly_sensitive** | boolean | Indique si le contenu peut être considéré comme sensible.                                                                           | Étudier la diffusion de certains types de contenu.                   |
| **promoted_metrics**  | object | Mesures d’engagement suivies dans un contexte promotionnel pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur. | Mesurer l’engagement du Tweet lorsqu’il a été promu.                 |
| **public_metrics**    | object | Mesures d’engagement publiques pour le Tweet au moment de la requête.                                                               | Mesurer l’engagement du Tweet.                                       |
| **referenced_tweets** | array  | Liste des Tweets auxquels ce Tweet fait référence, comme les Retweets, Tweets cités ou réponses.                                    | Comprendre les aspects conversationnels des retweets, etc.           |
| **reply_settings**    | string | Indique qui peut répondre à un Tweet donné. Les options sont « everyone », « mentioned_users » et « followers ».                    | Déterminer les paramètres de réponse de la conversation pour le Tweet. |
| **withheld**          | object | Contient les détails relatifs au [contenu restreint](https://help.x.com/en/rules-and-policies/tweet-withheld-by-country).         |                                                                     |
| **media_metadata**          | array | Contient les métadonnées des médias joints au Tweet.                                          | Obtenir des métadonnées supplémentaires, comme le `alt_text` d’un média joint à un Tweet. |

**Récupérer un objet Tweet**

**Exemple de requête**

Dans la requête suivante, nous demandons des champs pour le Tweet sur l’endpoint [Tweets lookup](/fr/resources/fundamentals/rate-limits). Assurez-vous de remplacer « $BEARER_TOKEN » par votre propre [jeton Bearer](/fr/resources/fundamentals/authentication) généré.

```bash
curl --request GET 'https://api.x.com/2/tweets?ids=1212092628029698048&tweet.fields=attachments,author_id,context_annotations,created_at,entities,geo,id,in_reply_to_user_id,lang,possibly_sensitive,public_metrics,referenced_tweets,text,withheld&expansions=referenced_tweets.id' --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="user">
  ### Utilisateur
</div>

L’objet utilisateur contient les métadonnées du compte Twitter décrivant l’utilisateur référencé. Il s’agit de l’objet principal renvoyé par le point de terminaison [users lookup](/fr/x-api/users/lookup/introduction). Pour demander des champs utilisateur supplémentaires sur ce point de terminaison, utilisez simplement le paramètre de champs `user.fields`.

L’objet utilisateur peut également apparaître comme objet enfant et être étendu dans l’objet Tweet. Il est disponible via l’extension `?expansions=author_id` ou `?expansions=in_reply_to_user_id` pour obtenir l’objet condensé avec uniquement les champs par défaut. Utilisez cette extension avec le paramètre de champs `user.fields` lorsque vous demandez des champs supplémentaires afin de compléter l’objet.
 

| Valeur du champ | Type | Description | Utilisation |
| :--- | :--- | :--- | :--- |
| id (par défaut) | string | Identifiant unique de cet utilisateur.<br /><br />`"id": "2244994945"` | À utiliser pour récupérer par programmation des informations sur un utilisateur Twitter spécifique. |
| name (par défaut) | string | Le nom de l’utilisateur tel qu’il l’a défini sur son profil. Pas nécessairement un nom de personne. Généralement plafonné à 50 caractères, mais susceptible d’évoluer.<br /><br />`"name": "Twitter Dev"` |     |

```{
  curl --request GET 'https://api.x.com/2/users?
  ids=2244994945&user.fields=created_at,description,entities,id,location,name,pinned_tweet_id,profile_image_url,protected,url,username,verified,withheld&expansions=pinned_tweet_id'
  --header 'Authorization: Bearer $BEARER_TOKEN'
  }
```

</div>

Les Spaces permettent de s’exprimer et d’interagir via des conversations audio en direct. Le dictionnaire de données des Spaces contient les métadonnées pertinentes d’un Space ; tous les détails sont mis à jour en temps réel.

<div id="direct-message-events">
  ### Événements de messages privés
</div>

Les conversations de messages privés (DM) sont constituées d’événements. L’API X v2 prend actuellement en charge trois types d’événements : MessageCreate, ParticipantsJoin et ParticipantsLeave.

Les objets d’événements DM sont renvoyés par les points de terminaison de [consultation des messages privés](/fr/x-api/direct-messages/lookup/introduction), et un événement MessageCreate est créé lorsque des messages privés sont créés avec succès via les points de terminaison de [gestion des messages privés](/fr/x-api/direct-messages/lookup/introduction).

Lors de la récupération d’événements DM, trois attributs par défaut de l’objet événement, ou champs, sont inclus : id, event_type et text. Pour recevoir des champs d’événements supplémentaires, utilisez le paramètre [fields](/fr/x-api/fundamentals/fields) dm_event.fields pour en sélectionner d’autres. Les autres champs d’événements disponibles incluent notamment : dm_conversation_id, created_at, sender_id, attachments, participant_ids et referenced_tweets.

Plusieurs de ces champs fournissent les identifiants d’autres objets X liés à l’événement de message privé :

* sender_id - L’identifiant du compte qui a envoyé le message, ou qui a invité un participant à une conversation de groupe
* participant_ids - Un tableau d’identifiants de comptes. Pour les événements ParticipantsJoin et ParticipantsLeave, ce tableau contient un seul identifiant : celui du compte qui a créé l’événement
* attachments - Fournit les identifiants de médias pour le contenu téléversé sur X par l’expéditeur
* referenced_tweets - Si une URL de Tweet est trouvée dans le champ text, l’identifiant de ce Tweet est inclus dans la réponse

Les [expansions](/fr/x-api/fundamentals/expansions) sender_id, participant_ids, referenced_tweets.id et attachments.media_keys sont disponibles pour développer ces identifiants d’objets X.

**Récupération d’un objet d’événement de Message privé**

**Exemple de requête**

Pour cet exemple, nous allons construire une requête qui récupère les événements associés à une conversation en tête-à-tête. * Des attributs d’événement fondamentaux, tels que quand il a été créé et à quelle conversation il appartient (dm_conversation).
* L’ID du compte et la description de la personne qui a envoyé le Message privé.
* Le texte de tout Tweet référencé, et quand il a été publié.
* L’ID du compte et la description de tout auteur de Tweet référencé.

<div id="how-to-use-fields-and-expansions">
  ## Comment utiliser les champs et les expansions
</div>

Par défaut, les objets de données de l’API X v2 incluent un petit nombre de champs prédéfinis lorsque vous effectuez une requête sans utiliser les paramètres [fields](/fr/x-api/fundamentals/fields) ou [expansions](/fr/x-api/fundamentals/expansions). Ce guide vous montrera comment utiliser les paramètres de requête `fields` et `expansions` afin de recevoir des objets et des champs supplémentaires dans votre réponse.

Dans ce guide, nous allons demander plusieurs champs à partir de la capture d’écran du Tweet ci-dessous.
 

![Cette image montre une capture d’écran d’un Tweet publié par @X. Vous pouvez voir le texte du Tweet, le nom d’utilisateur, la date et l’heure de publication, la source et les métriques publiques. Elle inclut également une vidéo. ](https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/ScreenShotDataDictionaryGuide.png.twimg.1920.png)

Comme vous pouvez le voir sur la capture d’écran, plusieurs informations visibles sont liées au Tweet, notamment l’auteur du Tweet, les métriques du Tweet, l’horodatage de création, la vidéo et le nombre de vues de la vidéo. D’autres données ne sont pas visibles sur la capture, mais restent disponibles sur demande. 

Lorsque vous effectuez une requête vers l’API, la réponse par défaut est simple et ne contient que les champs par défaut du Tweet (id et text). Vous ne recevrez également que l’objet principal renvoyé par le point de terminaison que vous utilisez, et non les objets de données associés pouvant être liés à cet objet principal.

Cette simplicité, associée aux paramètres fields et expansions, vous permet de demander uniquement les champs dont vous avez besoin, selon votre cas d’usage.

curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969' \
