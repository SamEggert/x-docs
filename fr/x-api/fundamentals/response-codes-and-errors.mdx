L’API X fournit les codes de réponse et d’erreur suivants pour faciliter la compréhension et le débogage en temps réel. Utilisez le guide de débogage et l’index des erreurs ci-dessous pour plus de contexte.

Les réponses réussies sont indiquées par un code HTTP de la série 200 et une charge utile JSON contenant l’objet ou les objets demandés, créés, modifiés ou supprimés, ainsi qu’une indication de l’interprétation de votre requête par le serveur.

Les réponses d’erreur sont renvoyées avec un code HTTP hors série 200. Des codes d’erreur différents signalent des causes d’erreur différentes. L’API X s’efforce de renvoyer les [codes d’état HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriés pour chaque requête.

<div id="x-api-v2-http-status-codes">
  ## Codes d’état HTTP de X API v2
</div>

| Code | Texte | Description | Conseils de dépannage |
| :--- | :--- | :---------- | :------------------- |
| 200 | OK | La requête a abouti. | |
| 400 | Bad Request | La requête est invalide ou ne peut pas être traitée. Un message d’erreur d’accompagnement apportera des précisions. Les requêtes sans authentification ou avec des paramètres de requête invalides sont considérées comme invalides et renvoient cette réponse. | Vérifiez le format de votre requête JSON. Par exemple, si votre règle contient des guillemets doubles associés à une correspondance exacte ou à un autre opérateur, vous devrez peut-être les échapper avec une barre oblique inversée pour les distinguer de la structure du format JSON. |
| 401 | Unauthorized | Un problème est survenu lors de l’authentification de votre requête. Cela peut être dû à des informations d’authentification manquantes ou incorrectes. Ce code peut également être renvoyé dans d’autres circonstances non définies. | Vérifiez que vous utilisez la bonne [méthode d’authentification](https://docs.x.com/resources/fundamentals/authentication/overview) et que vos informations d’identification sont correctes. |
| 403 | Forbidden | La requête est comprise, mais refusée, ou l’accès n’est pas autorisé. Un message d’erreur d’accompagnement expliquera pourquoi. | Vérifiez que votre compte développeur inclut l’accès à l’endpoint que vous essayez d’utiliser. Vous devrez peut-être aussi faire placer votre application sur liste d’autorisation (p. ex., Engagement API ou Ads API) ou demander l’accès. |
| 404 | Not Found | L’URI demandé est invalide ou la ressource demandée (par exemple, un utilisateur) n’existe pas. | Vérifiez que vous utilisez des paramètres valides et l’URI correct pour l’endpoint que vous utilisez. |
| 409 | Connection Exception | Renvoyé lorsque vous tentez de vous connecter à un flux filtré qui n’a aucune règle. | Vérifiez que vous avez créé au moins une règle sur le flux auquel vous vous connectez. Un flux filtré ne renvoie que les posts correspondant à une règle active. S’il n’y a aucune règle, le flux ne renverra aucun post. |
| 429 | Too Many Requests | Renvoyé lorsqu’une requête ne peut pas être servie parce que la [limite de débit](/fr/resources/fundamentals/rate-limits) de l’application ou le [plafond de publication](/fr/x-api/fundamentals/post-cap) est atteint. Voir [Limitation de débit](/fr/resources/fundamentals/rate-limits). | Vérifiez le nombre de requêtes autorisées par fenêtre temporelle pour l’endpoint utilisé. Attendez la réinitialisation de la fenêtre. Espacez vos requêtes pour éviter d’atteindre les limites ou passez au palier de données supérieur. |
| 500 | Internal Server Error | Une anomalie s’est produite. Il s’agit généralement d’une erreur temporaire, par exemple en cas de forte charge ou si un endpoint rencontre temporairement des problèmes. | Consultez la [page d’état](/fr/status) de X API ou le [forum de la communauté des développeurs](https://devcommunity.x.com/) pour vérifier si d’autres rencontrent des problèmes similaires, ou patientez et réessayez plus tard. |
| 501 | Unimplemented | X API ne prend pas en charge cet endpoint et ne peut pas satisfaire la requête. |  |
| 503 | Service Unavailable | Les serveurs X sont opérationnels mais surchargés de requêtes. Réessayez plus tard. | Consultez la [page d’état](/fr/status) de X API ou le [forum de la communauté des développeurs](https://devcommunity.x.com/) pour vérifier si d’autres rencontrent des problèmes similaires, ou patientez et réessayez plus tard. |

Lorsqu’une erreur survient pendant une requête, des informations détaillées sur l’erreur sont renvoyées dans le corps de la réponse pour aider à diagnostiquer le problème. Un champ type, qui est un URI, indique la nature du problème, tandis que des champs supplémentaires fournissent des précisions. Les champs type, title et detail sont toujours renvoyés dans ces corps (voir le tableau ci-dessous). Les champs supplémentaires, comme dans l’exemple ci-dessous, varient selon le type d’erreur.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "This request must be made using an approved developer account that is enrolled in the requested endpoint. Learn more by visiting our documentation.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Erreurs partielles
</div>

Dans certains cas, vous pouvez voir les erreurs détaillées ci‑dessus dans une réponse ayant renvoyé un code d’état 200. Dans ces cas, le point de terminaison est conçu pour renvoyer les données qu’il peut, tout en fournissant des erreurs détaillées sur ce qu’il n’a pas pu renvoyer.

Par exemple, le [point de terminaison de consultation des Posts](/fr/x-api/posts/lookup/introduction) permet à une [application développeur](/fr/resources/fundamentals/developer-apps#overview) X de demander plusieurs identifiants. Si certains de ces Posts sont disponibles mais que l’un d’eux a été supprimé, les Posts disponibles seront renvoyés dans le champ data de la réponse. Un champ errors supplémentaire sera inclus dans la charge utile, indiquant quel(s) Post(s) demandé(s) n’ont pas pu être renvoyé(s). Le même format que pour les erreurs de requête complètes est utilisé afin de faciliter le diagnostic des problèmes.

<div id="error-information">
  ### Informations sur les erreurs
</div>

Chaque type de problème précise la nature de l’erreur rencontrée. Vous trouverez également la liste complète des problèmes possibles dans notre [spécification de l’API](https://api.x.com/2/openapi.json).

L’API X tente de renvoyer des [codes d’état HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriés pour chaque requête.

| Titre | Type | Description |
| :--- | :--- | :--- |
| Problème générique | `about:blank` | Un problème générique sans information supplémentaire au-delà de celle fournie par le code d’état HTTP. |
| Problème de requête invalide | `https://api.X.com/2/problems/invalid-request` | Un problème indiquant que cette requête est invalide. Si votre requête inclut un corps POST, assurez-vous que le contenu est un JSON valide et qu’il correspond à la [spécification OpenAPI](https://api.x.com/2/openapi.json). |
| Problème de ressource introuvable | `https://api.X.com/2/problems/resource-not-found` | Un problème indiquant qu’une publication, un utilisateur, etc. n’existe pas. |
| Problème d’accès à la ressource non autorisé | `https://api.X.com/2/problems/not-authorized-for-resource` | Un problème indiquant que vous n’êtes pas autorisé à afficher une publication, un utilisateur, etc. particulier. |
| Problème client interdit | `https://api.X.com/2/problems/client-forbidden` | Un problème indiquant que votre client n’est pas autorisé à effectuer cette requête. |
| Problème de ressource non autorisée | `https://api.X.com/2/problems/disallowed-resource` | Un problème indiquant que la ressource demandée contrevient aux principes de cette API. |
| Problème d’authentification non prise en charge | `https://api.X.com/2/problems/unsupported-authentication` | Un problème indiquant que l’authentification utilisée n’est pas prise en charge. |
| Problème de plafond d’utilisation | `https://api.X.com/2/problems/usage-capped` | Un problème indiquant qu’un plafond d’utilisation a été dépassé. |
| Problème d’exception de connexion | `https://api.X.com/2/problems/streaming-connection` | Un problème indiquant un dysfonctionnement de la connexion. |
| Problème de client déconnecté | `https://api.X.com/2/problems/client-disconnected` | Votre client s’est déconnecté. |
| Problème de déconnexion opérationnelle | `https://api.X.com/2/problems/operational-disconnect` | Vous avez été déconnecté pour des raisons opérationnelles. |
| Problème de plafond de règles | `https://api.X.com/2/problems/rule-cap` | Vous avez dépassé le nombre maximal de règles. |
| Problème de longueur de règle | `https://api.X.com/2/problems/rule-length` | Vous avez dépassé le nombre maximal de caractères autorisés pour votre requête ou règle selon votre niveau d’accès. Voir [niveaux d’accès](/fr/x-api/getting-started/about-x-api#x-api-access-levels-and-versions). |
| Problème de règles invalides | `https://api.X.com/2/problems/invalid-rules` | La règle que vous avez soumise est invalide. |
| Problème de règles en double | `https://api.X.com/2/problems/duplicate-rules` | La règle que vous avez soumise est un doublon. |

<div id="troubleshooting-tips">
  ## Conseils de dépannage
</div>

<Accordion title="Débogage de votre code">
  Lors du développement d’une application, il est normal de rencontrer des erreurs ou des problèmes inattendus de temps à autre. Voici quelques conseils pour déboguer votre code :

  Commencez par décomposer le problème en éléments plus petits afin d’identifier où se situe l’erreur. Par exemple, si vous intégrez un endpoint REST ou de streaming, le problème peut provenir de l’un des points suivants :

  * L’endpoint requiert une [authentification](/fr/resources/fundamentals/authentication/overview) adéquate.
  * L’endpoint requiert des paramètres et des en-têtes valides. Toute règle de filtrage doit être construite avec les bons opérateurs et une [syntaxe](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) correcte.
  * L’URI de l’endpoint doit être correct et, pour les endpoints REST, la bonne méthode HTTP doit être utilisée.
  * Les données ou la ressource auxquelles vous tentez d’accéder ne vous sont pas accessibles (par exemple, les données privées ne sont disponibles que pour les utilisateurs authentifiés).
  * Votre forfait de données actuel ne donne accès qu’à certains endpoints et impose des limites de débit spécifiques. Consultez votre [portail développeur](https://developer.x.com/en/portal/products) pour plus de détails.
  * Votre code utilise une bibliothèque tierce pour intégrer l’endpoint.
  * Votre code doit analyser correctement la réponse de l’endpoint.

  Lisez le message d’erreur associé. Il devrait vous donner une bonne indication de la nature du problème. Utilisez les tableaux de la [section Codes d’erreur et de réponse](#x-api-http-status-codes) pour des conseils de dépannage propres à chaque code d’erreur.

  Pour les endpoints REST, vous pouvez utiliser un client REST comme [Postman](https://www.postman.com/) ou [Insomnia](https://insomnia.rest/) pour valider les étapes (1) à (5) ci-dessus (consultez notre guide [Bien démarrer avec Postman](/fr/tutorials/postman-getting-started)). Si la requête effectuée avec le client REST renvoie un code de statut 200 (succès), vous pouvez en déduire que le problème vient de votre code ou de la bibliothèque que vous utilisez, et non de la requête à l’endpoint elle‑même.

  Pour les endpoints de streaming, vous pouvez utiliser cURL (un outil en ligne de commande permettant d’envoyer ou de recevoir des requêtes en utilisant la syntaxe d’URL) afin de vérifier si le problème provient de la requête vers l’endpoint (étapes (1) à (5) ci-dessus) ou de votre code lui‑même (étapes (6) à (7) ci-dessus).
</Accordion>

<Accordion title="Problèmes d’authentification">
  * Vérifiez que vous utilisez la [méthode d’authentification](/fr/resources/fundamentals/authentication/overview) appropriée requise par l’endpoint. Vous pouvez l’identifier sur la page de référence API de l’endpoint.
  * Vérifiez que vos identifiants d’authentification sont corrects. Vous pouvez vérifier ou régénérer les clés et jetons de votre application dans la [section Apps](https://developer.x.com/en/portal/projects-and-apps) de votre tableau de bord développeur (sous « Details »).
  * Vérifiez que vous avez correctement [autorisé votre requête OAuth 1.0a](/fr/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) avec `oauth_nonce`, `oauth_signature` et `oauth_timestamp`.
  * Si le problème persiste, envisagez d’utiliser une bibliothèque OAuth, un client REST comme [Postman](https://www.postman.com/) ou [Insomnia](https://insomnia.rest/), ou [xurl](https://github.com/xdevplatform/xurl).

  Consultez notre [guide sur l’authentification](/fr/resources/fundamentals/authentication/overview) pour plus d’informations sur l’ensemble des points ci-dessus.
</Accordion>

<Accordion title="Problèmes de limitation de débit et de plafonnement des Posts">
  Une erreur 429 peut survenir si vous atteignez une limite de débit pour un point de terminaison donné, ou si vous atteignez un plafonnement des Posts.

  Si l’erreur spécifique « Too many requests » est renvoyée, vous avez atteint la limite de débit du point de terminaison. En d’autres termes, vous avez dépassé le nombre maximal de requêtes autorisées pour un point de terminaison sur une période donnée.

  Notez que les limites de débit sont définies au niveau de l’application et au niveau de l’utilisateur.

  * Le niveau **X App** indique le nombre de requêtes autorisées lors de l’utilisation d’OAuth 2.0 App-Only, où les limites de débit sont appliquées globalement pour l’ensemble de l’application. Par exemple, si une méthode autorise 15 requêtes par fenêtre de limitation, vous pouvez effectuer 15 requêtes par fenêtre pour le compte de votre X App. Cette limite est entièrement distincte de la limite au niveau utilisateur. En savoir plus dans notre guide sur [OAuth 2.0 App-Only](/fr/resources/fundamentals/authentication/oauth-2-0/application-only).
  * Le niveau **contexte utilisateur** indique le nombre de requêtes pouvant être effectuées par jeton d’accès utilisateur lors de l’utilisation d’OAuth 1.0a User Context ou d’OAuth 2.0 Authorization Code avec PKCE. Par exemple, si une méthode autorise 15 requêtes par fenêtre de limitation, elle autorise 15 requêtes par fenêtre et par jeton d’accès utilisateur. Consultez notre guide sur l’obtention des jetons d’accès d’un utilisateur avec [OAuth 1.0a](/fr/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) et [OAuth 2.0](/fr/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Commencez par vérifier les limites de débit du point de terminaison que vous utilisez. Vous trouverez ces informations sur la page de référence de l’API du point de terminaison et dans le nouveau tableau de bord du portail développeur.

  Consultez notre documentation pour plus d’informations sur les limites de débit, notamment comment utiliser les en-têtes HTTP pour suivre la position de votre application par rapport à une limite donnée, comment se remettre d’un code d’erreur 429 lié à la limitation de débit, ainsi que des conseils pour éviter d’être limité dès le départ :

  * [Limites de débit pour les points de terminaison v2](/fr/x-api/fundamentals/rate-limits)

  Si vous avez reçu l’erreur spécifique « Usage cap exceeded: Monthly product cap », cela signifie que vous avez atteint le plafonnement des Posts pour votre niveau d’accès. Nous détaillons ces [plafonnements de Posts](/fr/x-api/fundamentals/post-cap) sur notre page de documentation.
</Accordion>

<Accordion title="Problèmes de Post manquant">
  Suivez les étapes ci-dessous si vous vous attendiez à ce qu’un Post soit renvoyé, mais qu’il n’a pas été livré par le point de terminaison.

  * Vérifiez votre règle pour vous assurer que vous utilisez les [opérateurs](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) et la [syntaxe](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) appropriés. Décomposez la règle en clauses plus petites pour valider la syntaxe.
  * Si le compte depuis lequel le Post a été envoyé était [protégé](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) au moment de la création du Post, le Post ne sera pas renvoyé — même si le compte est public au moment de la requête adressée au point de terminaison. Vous pouvez généralement le vérifier avec la [recherche avancée X](https://x.com/search-advanced) : si un Post n’apparaît pas via la recherche avancée X, supposez qu’il ne sera pas renvoyé par le point de terminaison.

  Les étapes suivantes s’appliquent uniquement aux points de terminaison de streaming :

  * Étiez-vous connecté au flux lorsque le Post a été envoyé ? Rappelez-vous que l’horodatage fourni dans l’objet Post est en UTC. Si vous avez connu une déconnexion lorsque le Post a été envoyé, consultez les [fonctionnalités de reprise et de redondance](/fr/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponibles pour reconstituer les données manquées.
  * Votre règle était-elle en place lorsque le Post a été envoyé ? Rappelez-vous que l’horodatage fourni dans l’objet Post est en UTC.
</Accordion>

<Accordion title="Problèmes de déconnexion pour tampon plein">
  Vous pouvez rencontrer l’une des erreurs suivantes lorsque votre flux ne suit pas la cadence à laquelle nous diffusons les données et que votre application ne consomme pas les données du flux assez rapidement :

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Nous tolérons un retard de diffusion pendant un certain temps et disposons, de notre côté, d’un tampon temporaire pour chaque flux ; mais si vous ne rattrapez pas ce retard, nous initions une déconnexion afin que vous puissiez vous reconnecter au point courant. Veuillez noter que cela peut entraîner une perte de données (pour les données présentes dans le tampon au moment de la déconnexion pour tampon plein).

  Ces problèmes peuvent survenir lors de fortes pointes de volume de données. De manière générale, nous recommandons d’utiliser un processus de mise en file tampon, distinct du processus de traitement, pour consommer rapidement les données.

  Si vous êtes un client entreprise utilisant les endpoints v1.1, vous pouvez en savoir plus sur l’optimisation de votre application pour éviter ce type de déconnexions dans nos articles sur la [connexion](/fr/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) et sur la consommation de données en streaming [ici](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) et [ici](/fr/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  Il existe une série d’outils permettant de récupérer les Posts manqués en raison d’une déconnexion, notamment ceux listés ci-dessous. Notez que les outils suivants ne sont disponibles qu’avec les endpoints v1.1 au niveau d’accès entreprise.

  * [**Redundant Connections**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Avec plusieurs connexions, consommez le flux depuis plusieurs serveurs pour éviter les données manquées lorsqu’une connexion est interrompue.
  * [**Replay**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Récupérez les données des 5 derniers jours à l’aide d’un flux séparé.
  * [**Backfill**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Reconnectez-vous dans les 5 minutes et reprenez là où vous vous êtes arrêté.
  * **Historical PowerTrack** - Récupérez des données de l’intégralité des archives X.
</Accordion>

<div id="before-you-post-a-question">
  ### Avant de publier une question
</div>

* Recherchez dans la documentation des développeurs X des informations liées à votre problème
* Recherchez sur le forum de la communauté des questions similaires posées par d’autres développeurs
* Consultez les règles du forum de la communauté

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Lorsque vous publiez une question, veillez à inclure les informations suivantes
</div>

* Une description du problème
* L’appel à l’API effectué (incluez les en-têtes, si possible)
* La réponse X renvoyée (incluez tout message d’erreur)
* Ce que vous vous attendiez à recevoir à la place
* La liste des étapes effectuées pour résoudre le problème
* La liste des étapes nécessaires pour reproduire le problème
* Le cas échéant, la période durant laquelle le problème s’est produit
* Le cas échéant, l’ID d’application, l’ID de post, etc.
* Tout extrait de code ou toute capture d’écran pertinent(e)

Veuillez n’inclure qu’un seul sujet ou une seule question par post.

Si vous avez des demandes de fonctionnalités ou des commentaires, veuillez les soumettre via le [formulaire de retour d’expérience de la plateforme développeur X](https://t.co/devfeedback).

Pour les problèmes liés aux règles, tels que la suspension d’une application, veuillez contacter le [support des règles](https://help.twitter.com/forms/platform).

Pour les problèmes liés à X, tels que la connexion et l’assistance relative au compte, veuillez utiliser le [Centre d’aide X](https://help.twitter.com/en/contact-us).
