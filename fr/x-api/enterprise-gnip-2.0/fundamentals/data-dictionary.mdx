<div id="introduction">
  ### Introduction
</div>

`Enterprise`

Les Posts sont l’unité atomique de base de tout ce qui concerne X. Toutes les API X qui renvoient des Posts fournissent ces données encodées en JavaScript Object Notation (JSON). Le JSON repose sur des paires clé-valeur, avec des attributs nommés et des valeurs associées. Les objets Post récupérés via l’API incluent la « mise à jour de statut » d’un utilisateur X, mais les Retweets, les réponses et les Tweets cités sont également des objets Post. Si un Post est lié à un autre Post, comme un Retweet, une réponse ou un Tweet cité, chacun sera identifié ou imbriqué dans l’objet Post. Même le Post le plus simple, au format de données natif de X, comportera des objets JSON imbriqués pour représenter d’autres attributs d’un Post, tels que l’auteur, les utilisateurs mentionnés, le lieu tagué, les hashtags, les symboles de cashtag, les médias ou les liens URL. Lors de l’utilisation des données X, c’est un concept important à comprendre. Le format des données de Post que vous recevrez de l’API X dépend du type de Post, de l’API X utilisée et des paramètres de format.

Les endpoints Enterprise qui renvoient des objets Post ont été mis à jour pour fournir les métadonnées nécessaires à la compréhension de l’historique des modifications du Post. Pour en savoir plus sur ces métadonnées, consultez la page [&quot;Modifier des Posts&quot;](/fr/x-api/fundamentals/edit-posts).

Dans le format natif X, la charge utile JSON inclut des attributs de niveau racine et des objets JSON imbriqués (représentés ici par la notation `{}`) :

```JSON
{
	"created_at": "Fri Feb 14 19:00:55 +0000 2020",
	"id_str": "1228393702244134912",
	"text": "Qu'est-ce que le développeur a écrit dans sa carte de Saint-Valentin ?\n  \nwhile(true) {\n    I = Love(You);  \n}",
	"entities": {
		"hashtags": [],
		"symbols": [],
		"user_mentions": [],
		"urls": []
	},
	"user": {
		"entities": {
			"url": {}
		}
	},
	"place": {}
}
```

<div id="available-data-formats">
  ### Formats de données disponibles
</div>

> Veuillez noter : il est fortement recommandé d&#39;utiliser le format [Native Enriched](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) pour les API de données Enterprise.
>
> * Le format Native Enriched inclut toutes les nouvelles métadonnées depuis 2017, telles que les [métadonnées de sondage](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), ainsi que des métriques supplémentaires comme reply_count et quote_count.
>
> * Le format Activity Streams n&#39;a pas été mis à jour avec de nouvelles métadonnées ou de nouveaux enrichissements depuis la [mise à jour des caractères](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) de 2017.

Les API de données Enterprise fournissent des données dans deux formats différents. Le format Enterprise le plus proche du format natif standard v1.1 est Native Enriched. L&#39;ancien format de données Enterprise est Activity Streams, initialement implémenté et utilisé par Gnip comme format normalisé pour X et d&#39;autres fournisseurs de données de médias sociaux à l&#39;époque. Bien que ce format soit toujours disponible, X n&#39;a investi dans de nouvelles fonctionnalités et évolutions que pour le format Native Enriched depuis 2017.

Le format Native Enriched est exactement ce que son nom suggère : il inclut les objets natifs X ainsi que des enrichissements supplémentaires disponibles pour les produits de données Enterprise, tels que les métadonnées de déroulage d&#39;URL, le profil géographique, les métadonnées de sondage et des métriques d&#39;engagement supplémentaires.

<div id="parsing-best-practices">
  ### Bonnes pratiques d’analyse
</div>

* Le JSON de X est encodé en UTF-8.
* Les analyseurs doivent tolérer sans difficulté des variations dans l’ordre des champs. On considérera que le JSON de Post est servi comme une table de hachage non ordonnée.
* Les analyseurs doivent tolérer l’ajout de nouveaux champs.
* Les analyseurs JSON doivent tolérer l’absence de certains champs, car tous les champs n’apparaissent pas dans tous les contextes.
* Il est généralement acceptable de considérer qu’un champ nul, un ensemble vide et l’absence d’un champ sont équivalents.

<div id="post-object">
  #### Objet Post
</div>

Lorsque vous utilisez les produits de données Enterprise, vous constaterez qu’une grande partie du dictionnaire de données est similaire au format natif des données Post, avec quelques métadonnées enrichies supplémentaires. Le niveau de base du format natif enrichi reprend bon nombre des mêmes noms d’objet que le [format de données X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). L’objet Post comporte une longue liste d’attributs « au niveau racine », notamment des attributs fondamentaux tels que `id`, `created_at` et `text`. Les objets Post comportent également des objets imbriqués, dont `user`, `entities` et `extended_entities`. Les objets Post incluent aussi d’autres [objets Post imbriqués](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects) tels que retweeted_status, quoted_status et extended_tweet. Le format natif enrichi comporte en outre un objet matching_rules.

<div id="nested-post-objects">
  #### Objets Post imbriqués
</div>

Dans plusieurs cas, un objet Post inclut d’autres objets imbriqués. Si vous travaillez avec des objets imbriqués, la charge utile JSON contiendra alors plusieurs objets Post, et chaque objet Post peut contenir ses propres objets. L’objet racine contient des informations sur le type d’action effectuée, c’est‑à‑dire s’il s’agit d’un Retweet ou d’un Quote Tweet, et peut aussi contenir un objet décrivant le Post « original » partagé. Les Extended Posts incluent un objet étendu imbriqué qui dépasse 140 caractères, utilisé afin d’éviter des changements incompatibles lors de la mise à jour de 2017. Chaque dictionnaire d’objet imbriqué est décrit ci‑dessous.

**Retweets**

Les Retweets contiennent toujours deux objets Post. Le Post « original » retweeté est fourni dans un objet &quot;retweeted_status&quot;. L’objet racine encapsule le Retweet lui‑même, y compris un objet User pour le compte effectuant l’action de Retweet et l’heure du Retweet. Le Retweet est une action permettant de partager un Post avec vos abonnés, sans possibilité d’ajouter de nouveau contenu. De plus, il est impossible de fournir une (nouvelle) localisation avec un Retweet. Même si le Post « original » peut être géolocalisé, les objets &quot;geo&quot; et &quot;place&quot; du Retweet seront toujours null.

Même avant l’introduction des Extended Posts, l’objet &quot;entities&quot; au niveau racine était dans certains cas tronqué et incomplet en raison de l’ajout de la chaîne &quot;RT @username &quot; au message du Post retweeté. Notez que si un Retweet est à son tour retweeté, &quot;retweet_status&quot; pointera toujours vers le Post original, ce qui signifie que le Retweet intermédiaire n’est pas inclus. Un comportement similaire est observé lors de l’utilisation de x.com pour « afficher » un Retweet. Si vous copiez l’ID de Post unique attribué à l’« action » de Retweet, le Post original est affiché.

Ci‑dessous, un exemple de structure pour un Retweet. Là encore, lors de l’analyse des Retweets, il est essentiel d’extraire l’objet &quot;retweeted_status&quot; pour obtenir le message de Post complet (original) et les métadonnées des entités.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

<div id="quote-tweets">
  ##### Tweets cités
</div>

Les Tweets cités ressemblent beaucoup aux Retweets, à ceci près qu’ils incluent un nouveau message de Post. Ces nouveaux messages peuvent contenir leur propre ensemble de hashtags, de liens et d’autres métadonnées « entities ». Les Tweets cités peuvent aussi inclure des informations de localisation partagées par l’utilisateur qui publie le Tweet cité, ainsi que des médias tels que des GIF, des vidéos et des photos.

Les Tweets cités contiennent au moins deux objets Post, et dans certains cas trois. Le Post cité, qui peut lui-même être un Tweet cité, est fourni dans un objet « quoted_status ». L’objet de niveau racine encapsule le Tweet cité lui-même, y compris un objet User pour le compte effectuant l’action de partage et l’horodatage du Tweet cité.

Notez que les Tweets cités peuvent désormais inclure des photos, des GIF ou des vidéos, ajoutés via l’interface utilisateur « Post ». Lorsque des liens vers des médias hébergés en externe sont inclus dans le message du Tweet cité, le « entities.urls » de niveau racine les décrira. Les médias joints aux Tweets cités apparaîtront dans les métadonnées « extended_entities » de niveau racine.

Au lancement des Tweets cités, un lien raccourci (URL t.co) était ajouté au message du Post « original » et fourni dans le champ « text » de niveau racine. De plus, les métadonnées de cette URL t.co étaient incluses dans le tableau « entities.urls » de niveau racine. En mai 2018, nous avons modifié ce comportement : l’URL t.co raccourcie vers le Tweet cité n’est plus incluse dans le champ « text » de niveau racine. Ensuite, les métadonnées du Tweet cité ne sont plus incluses dans les métadonnées « entities.urls ». À la place, les métadonnées d’URL du Tweet cité se trouvent dans un nouvel objet « quoted_status_permalink » au niveau racine (ou niveau supérieur), donc au même niveau que l’objet « quoted_status ».

<div id="extended-posts">
  ##### Publications étendues
</div>

Le JSON décrivant les *Publications étendues* a été introduit lors du lancement des Publications de 280 caractères en novembre 2017. Le JSON de Publication a été étendu pour encapsuler ces messages plus longs, sans rompre la compatibilité avec les milliers d’applications qui analysent ces objets X fondamentaux. Afin d’assurer une rétrocompatibilité totale, le champ « text » d’origine, limité à 140 caractères, ainsi que les objets d’entités qui en sont extraits, ont été conservés. Pour les Publications dépassant 140 caractères, ce champ « text » au niveau racine est tronqué et donc incomplet. Étant donné que les objets « entities » au niveau racine contiennent des tableaux de métadonnées clés extraites du message « text » (comme les hashtags et les liens inclus), ces collections seraient incomplètes. Par exemple, si un message de Publication faisait 200 caractères, avec un hashtag à la fin, le tableau hérité « entities.hashtags » au niveau racine ne l’inclurait pas.

Un nouveau champ « extended_tweet » a été introduit pour contenir les messages de Publication plus longs ainsi que des métadonnées d’entités complètes. L’objet « extended_tweet » fournit le champ « full_text », qui contient le message de Publication complet, non tronqué, lorsqu’il dépasse 140 caractères. L’objet « extended_tweet » contient également un objet « entities » avec des tableaux complets de hashtags, liens, mentions, etc.

Les Publications étendues sont identifiées par un booléen « truncated » au niveau racine. Lorsque true (« truncated »: true), il faut analyser les champs « extended_tweet » plutôt que les champs au niveau racine.

Notez dans l’exemple JSON ci-dessous que le champ « text » au niveau racine est tronqué et que le tableau « entities.hashtags » au niveau racine est vide alors même que le message de la Publication contient trois hashtags. Étant donné qu’il s’agit d’une Publication étendue, le champ « truncated » est défini sur true, et l’objet « extended_tweet » fournit des métadonnées complètes de Publication pour « full_text » et « entities ».

<div id="native-enriched-geo-objects">
  ### Objets Geo natifs enrichis
</div>

Les Posts peuvent être associés à un lieu, générant un Post « géolocalisé ». Les lieux peuvent être attribués via l’interface utilisateur de X ou lors de la publication d’un Post via l’API. Le lieu d’un Post peut être un « point » exact, ou un X Place avec une « boîte englobante » décrivant une zone plus vaste, allant d’un établissement à une région entière.

Il existe trois objets JSON « racine » utilisés pour décrire le lieu associé à un Post : [place](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) et [coordinates](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary).

De plus, le format natif enrichi inclut le [lieu déduit](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) de l’[enrichissement de géolocalisation du profil](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) dans l’objet utilisateur.

L’objet `place` est toujours présent lorsqu’un Post est géolocalisé avec un lieu. Les Places sont des emplacements précis et nommés, avec des coordonnées géographiques correspondantes. Lorsque les utilisateurs choisissent d’attribuer un lieu à leur Post, une liste de X Places candidates leur est proposée. Lors de la publication via l’API, un X Place peut être associé en spécifiant un place_id lors de la publication. Les Posts associés à des Places ne sont pas nécessairement émis depuis ce lieu, mais peuvent aussi potentiellement *concern­er* ce lieu.

Les objets `geo` et `coordinates` ne sont présents (non nuls) que lorsque le Post est associé à un *lieu exact*. Si un lieu exact est fourni, l’objet `coordinates` fournit un tableau [long, lat] avec les coordonnées géographiques, et un X Place correspondant à ce lieu est attribué.

<div id="introduction">
  ## Introduction
</div>

Les entités fournissent des métadonnées et des informations contextuelles supplémentaires sur le contenu publié sur X. La section `entities` contient des tableaux d’éléments courants inclus dans les Posts : hashtags, mentions d’utilisateurs, liens, tickers boursiers (symboles), sondages X et médias associés. Ces tableaux sont pratiques pour les développeurs lors de l’ingestion de Posts, puisque X a en quelque sorte prétraité, ou « pré‑parsé », le corps du texte. Plutôt que de devoir rechercher et identifier explicitement ces entités dans le corps du Post, votre analyseur peut accéder directement à cette section JSON où elles sont déjà présentes.

Au‑delà de ces facilités de parsing, la section `entities` fournit aussi des métadonnées à forte valeur ajoutée. Par exemple, si vous utilisez l’[enrichissement Enhanced URLs](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls), les métadonnées d’URL incluent les URL entièrement développées, ainsi que les titres et descriptions des sites web associés. Autre exemple : lorsqu’il y a des mentions d’utilisateurs, les métadonnées des entités incluent l’identifiant utilisateur numérique, utile lors de requêtes vers de nombreuses API X.

Chaque payload JSON de Post inclut une section `entities`, avec le jeu minimal d’attributs `hashtags`, `urls`, `user_mentions` et `symbols`, même si aucune de ces entités ne figure dans le message du Post. Par exemple, si vous examinez le JSON d’un Post dont le corps est « Hello World! » et sans média associé, le JSON du Post inclura le contenu suivant avec des tableaux d’entités vides :

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Notes :

* les entités media et polls n’apparaissent que lorsque ce type de contenu fait partie du Post.
* si vous travaillez avec des médias natifs (photos, vidéos ou GIF), l’[objet Extended Entities](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) est recommandé.

<div id="entities-object">
  ### Objet Entities
</div>

Les sections `entities` et `extended_entities` sont toutes deux constituées de tableaux d’*objets* d’entité. Vous trouverez ci-dessous des descriptions de chacun de ces objets d’entité, accompagnées de dictionnaires de données décrivant les noms des attributs de l’objet, leurs types et une courte description. Nous indiquerons également quels opérateurs PowerTrack correspondent à ces attributs et fournirons quelques exemples de charges utiles JSON.

Une collection d’entités courantes présentes dans les Posts, notamment des hashtags, des liens et des mentions d’utilisateur. L’objet `entities` inclut un attribut `media`, mais sa mise en œuvre dans la section `entities` n’est pleinement exacte que pour les Posts contenant une seule photo. Pour les Posts avec plusieurs photos, une vidéo ou un GIF animé, veuillez vous reporter à la section `extended_entities`.

<div id="photo-media-url-formatting">
  ### Formatage des URL des médias photo
</div>

Les médias photo sur X peuvent être chargés dans différentes tailles. Il est préférable de charger l’image la plus petite qui soit suffisamment grande pour s’adapter à une zone d’affichage donnée. Pour charger différentes tailles, le [Size Object](#size) et [media_url](#media) (ou [media_url_https](#media)) doivent être combinés selon un format particulier. Nous utiliserons l’exemple d’[entité media](#entitiesobject) déjà fourni pour construire une URL de média photo.

Les champs `media_url` ou `media_url_https` peuvent être chargés tels quels, ce qui entraînera le chargement par défaut de la variante medium. Il est toutefois préférable de fournir une URL de média photo entièrement formatée lorsque c’est possible.

Il y a trois parties dans une URL de média photo :

|     |     |
| :--- | :--- |
| URL de base | L’URL de base est l’URL du média sans l’extension de fichier.<br /><br />Par exemple :  <br /><br />&quot;media_url_https&quot;: &quot;https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg&quot;,<br /><br />L’URL de base est donc :  <br /><br />*https://pbs.twimg.com/media/DOhM30VVwAEpIHq* |
| Format | Le format correspond au type de fichier de la photo. Les formats possibles sont *jpg* ou *png*, fournis en tant qu’extension de l’URL du média.<br /><br />Par exemple :  <br /><br />&quot;media_url_https&quot;: &quot;https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg&quot;,<br /><br />Le format est donc : *jpg* |
| Nom | Le nom est le champ indiquant la taille à charger.<br /><br />Par exemple :<br /><br />  <br /> &quot;sizes&quot;:   <br />   &quot;thumb&quot;:   <br />     &quot;h&quot;: 150,  <br />     &quot;resize&quot;: &quot;crop&quot;,  <br />     &quot;w&quot;: 150  <br />   ,  <br />   &quot;large&quot;: <br />     &quot;h&quot;: 1366,  <br />     &quot;resize&quot;: &quot;fit&quot;,  <br />     &quot;w&quot;: 2048  <br />   &#125;,  <br />   &quot;medium&quot;: <br />     &quot;h&quot;: 800,  <br />     &quot;resize&quot;: &quot;fit&quot;,  <br />     &quot;w&quot;: 1200  <br />   &#125;,  <br />   &quot;small&quot;: <br />     &quot;h&quot;: 454,  <br />     &quot;resize&quot;: &quot;fit&quot;,  <br />     &quot;w&quot;: 680  <br />   &#125;  <br /> &#125;  <br />&#125;<br /><br />Le nom lors du chargement de la photo en grande taille serait : *large* |

Nous prenons ces trois parties (URL de base, format et nom) et les combinons pour former l’URL du média photo à charger. Il existe deux formats pour charger des images de cette manière : *legacy* et *modern*. Tous les chargements d’images doivent cesser d’utiliser le format *legacy* au profit du format *modern*. L’utilisation du format *modern* se traduira par un meilleur taux de succès de mise en cache CDN pour l’appelant, améliorant ainsi les temps de chargement en réduisant la probabilité d’avoir à générer et charger le média depuis le centre de données.

|     |     |
| :--- | :--- |
| Legacy format | Le format hérité est obsolète. Tous les chargements de médias photo doivent passer au format moderne.<br /><br />*&lt;base_url&gt;.&lt;format&gt;:&lt;name&gt;*  <br /><br />Par exemple :  <br /><br />*https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />* |
| Modern format | Le format moderne pour le chargement des photos a été introduit chez X en 2015 et s’est imposé de facto depuis 2017. Tous les chargements de médias photo doivent adopter ce format.<br /><br />*&lt;base_url&gt;?format=&lt;format&gt;&amp;name=&lt;name&gt;*  <br /><br />Par exemple :  <br /><br />*https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg&amp;name=large  <br />*<br /><br />Remarque : les éléments de la chaîne de requête de l’URL du média photo sont classés par ordre alphabétique. Si le chargement de médias devait ajouter des éléments de requête supplémentaires, le tri alphabétique resterait requis. Par exemple, si un nouvel élément de requête hypothétique appelé *preferred_format* était ajouté, il viendrait après *format* et *name* dans la chaîne de requête. |

<div id="retweet-and-quote-tweet-details">
  ## Détails sur les Retweets et les Tweets cités
</div>

Du point de vue de l’API X, les Retweets et les Tweets cités sont des types particuliers de publications qui contiennent la publication originale en tant qu’objet intégré. Ainsi, les objets Retweet et Tweet cité sont les parents d’une publication enfant « originale » (et sont donc deux fois plus volumineux). Les Retweets possèdent un objet de niveau supérieur &quot;retweeted_status&quot;, et les Tweets cités un objet &quot;quoted_status&quot;. Par cohérence, ces objets Retweet et Tweet cité de niveau supérieur disposent également d’une propriété text et d’entities associées. Toutefois, les entities au niveau supérieur peuvent différer des entities fournies par les entities « originales » intégrées. Dans le cas des Retweets, un nouveau texte est ajouté en préfixe au corps de la publication originale. Pour les publications citées, un nouveau texte est ajouté en suffixe au corps de la publication.

De manière générale, la meilleure pratique consiste à récupérer le texte, les entities, l’auteur original et la date à partir de la publication originale dans retweeted_status lorsque celui-ci existe. Une exception concerne la récupération des entities X qui font partie de l’ajout de la citation. Voir ci-dessous pour plus de détails et de conseils.

<div id="retweets">
  ### Retweets
</div>

Un point important concernant les Retweets est qu’aucune *entity* X supplémentaire ne peut être ajoutée à la publication. Les utilisateurs ne peuvent pas ajouter de hashtags, d’URL ou d’autres éléments lorsqu’ils retweetent. Toutefois, l’attribut text (de niveau supérieur) du Retweet est constitué du texte de la publication d’origine avec « RT @username: » ajouté en préfixe.  

Dans certains cas, notamment avec des comptes dont le nom d’utilisateur est long, la combinaison de ces nouveaux caractères et du corps de la publication d’origine peut facilement dépasser la limite de 140 caractères du texte de la publication. Afin de préserver la compatibilité avec l’affichage et le stockage basés sur 140 caractères, le corps de niveau supérieur tronque la fin du corps de la publication et ajoute une ellipse (« … »). Par conséquent, certaines entités de niveau supérieur positionnées à la fin de la publication d’origine peuvent être incorrectes ou manquantes, par exemple en cas de hashtag ou d’URL tronqués.

Cette publication,  https://x.com/FloodSocial/status/907974220298125312, contient le texte suivant :

<div id="quote-tweets">
  ### Tweets cités
</div>

Les Tweets cités ont été introduits en 2016 et diffèrent des Retweets en ce que, lorsque vous « citez » une Publication, vous ajoutez du nouveau contenu « par-dessus » une Publication partagée. Ce nouveau contenu peut inclure presque tout ce qu’une Publication originale peut contenir, y compris du nouveau texte, des hashtags, des mentions et des URL.

Les Tweets cités peuvent contenir des médias natifs (photos, vidéos et GIF) et apparaissent dans l’objet entities.

Étant donné que des entités X peuvent être ajoutées, les entités de la citation sont probablement différentes des entités originales.

Dans cet exemple, une nouvelle URL et un hashtag ont été placés à la fin du Tweet cité.

<div id="entities-for-user-object">
  ### Entités pour l'objet utilisateur
</div>

Les entités pour les objets utilisateur décrivent les URL qui apparaissent dans l’URL de profil définie par l’utilisateur et dans les champs de description. Elles ne décrivent pas les hashtags ni les user_mentions. Contrairement aux entités de publication, les entités utilisateur peuvent s’appliquer à plusieurs champs au sein de leur objet parent — pour lever toute ambiguïté, vous trouverez des nœuds parents nommés url et description qui indiquent quel champ contient l’URL entité.

<div id="introduction">
  #### Introduction
</div>

Si une Publication contient des médias natifs (partagés via l’interface utilisateur de la Publication plutôt que via un lien externe), une section extended_entities sera également présente. Pour tout média natif (photo, vidéo ou GIF), extended_entities est la source de métadonnées à privilégier pour plusieurs raisons. Actuellement, jusqu’à quatre photos peuvent être jointes à une Publication. Les métadonnées entities ne contiendront que la première photo (jusqu’en 2014, une seule photo pouvait être incluse), tandis que la section extended_entities inclura toutes les photos jointes. Avec les médias natifs, une autre limite des métadonnées entities.media est que le type de média indiquera toujours « photo », même lorsque le média joint est une vidéo ou un GIF animé. Le type réel du média est spécifié dans l’attribut extended_entities.media[].type et prend la valeur *photo*, *video* ou *animated_gif*. Pour ces raisons, si vous travaillez avec des médias natifs, les métadonnées extended_entities sont à privilégier.

Toutes les Publications comportant des photos, des vidéos ou des GIF animés joints incluront un objet JSON `extended_entities`. L’objet `extended_entities` contient un unique tableau `media` d’objets `media` (voir la section `entities` pour son dictionnaire de données). Aucun autre type d’entité, tel que les hashtags ou les liens, n’est inclus dans la section `extended_entities`. L’objet `media` dans la section `extended_entities` est identique, dans sa structure, à celui inclus dans la section `entities`.

Les Publications ne peuvent avoir qu’un seul type de média joint. Pour les photos, jusqu’à quatre photos peuvent être jointes. Pour les vidéos et les GIF, une seule peut être jointe. Étant donné que la métadonnée `type` du média dans la section `extended_entities` indique correctement le type de média (« photo », « video » ou « animated_gif ») et prend en charge jusqu’à 4 photos, il s’agit de la source de métadonnées recommandée pour les médias natifs.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

Lorsqu’un annonceur choisit de limiter la lecture vidéo aux seules plateformes appartenant à X et exploitées par elle, l’objet `video_info` est remplacé par un objet `additional_media_info`.

L’objet `additional_media_info` contient des informations média supplémentaires fournies par l’éditeur, telles que `title`, `description` et le « embeddable flag ». Le contenu vidéo n’est accessible qu’aux clients officiels de X lorsque `embeddable=false`. Dans ce cas, toutes les URL vidéo fournies dans la charge utile seront hébergées par X, afin que l’utilisateur puisse ouvrir la vidéo sur une propriété appartenant à X en cliquant sur le lien.

<div id="enterprise-activity-streams-data-objects">
  ## Objets de données Activity Streams pour l’entreprise
</div>

<Note>
  Vous souhaitez en savoir plus sur la correspondance entre le format de données Activity Streams et le format X API v2 ?
</Note>

Consultez notre guide comparatif : [Activity Streams comparé à X API v2](/fr/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  Remarque : il est fortement recommandé d’utiliser le format [Enriched Native](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) pour les API de données destinées aux entreprises.

  * Le format Enriched Native inclut toutes les métadonnées ajoutées depuis 2017, telles que les [métadonnées de sondage](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), ainsi que des métriques supplémentaires comme reply_count et quote_count.

  * Le format Activity Streams n’a pas été mis à jour avec de nouvelles métadonnées ou enrichissements depuis la [mise à jour du nombre de caractères](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) en 2017.
</Note>

<div id="activity-object">
  ### Objet Activity
</div>

Activity Streams est une traduction de schéma d’objet du [format de données d’origine](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) de X, créée par Gnip pour « normaliser le format » des données de Post et d’autres données de médias sociaux à l’aide du [schéma de base Activity décrit ici](https://activitystrea.ms/head/activity-schema.html) fourni par un tiers. Les Posts sont normalisés selon le schéma Activity Streams, notamment avec les types d’objets note, person, place et service, utilisés comme objets imbriqués. Les Posts peuvent contenir d’autres objets d’activité de Post imbriqués pour les Retweets, ou d’autres, notamment twitter_quoted_status, long_object.

Le type d’objet de base « activity » est similaire au [Post base level object](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) du format natif enrichi. Des exemples de charges utiles au format Activity Streams sont disponibles [ici](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

<div id="nested-post-activity-obejcts">
  #### Objets d’activité de publication imbriqués
</div>

Dans plusieurs cas, un objet Publication inclut d’autres publications imbriquées. Si vous travaillez avec des objets imbriqués, la charge utile JSON contiendra alors plusieurs objets, et chaque objet Publication pourra contenir ses propres objets. L’objet racine contiendra des informations sur le type d’action effectuée, c’est‑à‑dire s’il s’agit d’un Retweet ou d’un Tweet cité, et pourra également contenir un objet décrivant la publication « originale » partagée. Les publications étendues incluent un objet étendu imbriqué qui dépasse 140 caractères, utilisé pour éviter des changements incompatibles lors de la mise à jour de 2017. Chaque dictionnaire d’objet imbriqué est décrit ci‑dessous.

**Retweets**

Le format Activity Streams des Retweets inclut un objet imbriqué avec le type &quot;activity&quot; et le verbe &quot;note&quot; pour représenter la publication originale faisant l’objet du Retweet.

L’objet acteur contient des métadonnées de compte d’utilisateur X décrivant l’utilisateur X qui a créé l’activité.

<div id="data-dictionary">

</div>

| Attribut | Type | Description |
| :--- | :--- | :--- |
| objectType | string | **&quot;objectType&quot;:** &quot;person&quot; |
| id  | string | La représentation sous forme de chaîne de l’identifiant unique de cet auteur. Exemple :<br /><br />&quot;id:x.com:2244994945&quot; |

| preferredUsername | string | Le nom d’écran, handle ou alias avec lequel cet utilisateur s’identifie. Unique mais susceptible de changer. <div id="location-object">
  ### Objet Location
</div>

Les objets Location peuvent exister dans l’objet actor défini au niveau du compte X ou dans l’objet profileLocations de l’[objet gnip](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object). Les objets Location sont similaires à [Geo](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) au format natif enrichi.

<div id="location-data-dictionary">

### Introduction**

Au cœur de son fonctionnement, X est un réseau de communication public, en temps réel et mondial. Depuis 2006, l’évolution de X a été guidée à la fois par les usages et conventions des utilisateurs et par de nouvelles fonctionnalités et améliorations produit. Si vous utilisez des données X pour des recherches historiques, comprendre la chronologie de cette évolution est important pour faire ressortir, à partir des archives, les Posts présentant un intérêt.

X a été lancé comme une simple application mobile par SMS et est devenu une plateforme de communication complète. Une plateforme avec un ensemble complet d’API. Les API ont toujours été un pilier du réseau X. La [première API a vu le jour peu après le lancement de X](https://blog.x.com/2006/introducing-the-twitter-api). Lorsque l’ajout de balises de localisation aux Posts a été introduit pour la première fois en 2009, il a été proposé via une [API Geo](https://blog.x.com/2009/think-globally-tweet-locally) (et plus tard, la possibilité de « géolocaliser » un Post a été intégrée à l’interface de X.com). Aujourd’hui, les API de X alimentent le réseau de communication bidirectionnel devenu une source d’actualités de dernière minute et de partage d’informations. Les opportunités de construire sur ce canal de communication mondial et en temps réel sont infinies.

X met à disposition deux API historiques qui offrent un accès à chaque Post public : Historical PowerTrack et la [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Les deux API fournissent un ensemble d’*opérateurs* utilisés pour interroger et collecter les Posts d’intérêt. Ces opérateurs s’appuient sur une variété d’attributs associés à chaque Post — des centaines d’attributs, tels que le contenu textuel du Post, le nom de compte de l’auteur et les liens partagés dans le Post. Les Posts et leurs attributs sont encodés en JSON, un format d’échange de données textuel courant. Ainsi, à mesure que de nouvelles fonctionnalités étaient introduites, de nouveaux attributs JSON apparaissaient et, en général, de nouveaux opérateurs d’API étaient ajoutés pour cibler ces attributs. Si votre cas d’usage implique d’*écouter* ce que le monde a dit sur X, plus vous comprendrez à partir de quand les opérateurs ont disposé de métadonnées JSON sur lesquelles s’appuyer, plus vos filtres Historical PowerTrack seront efficaces.

Ensuite, nous présenterons quelques concepts clés qui prépareront le terrain pour comprendre comment les mises à jour des métadonnées des Posts influent sur l’identification de votre signal de données d’intérêt.

### Concepts clés**

<div id="from-user-conventions-to-x-first-class-objects">
  #### Des conventions des utilisateurs aux *objets de première classe* de X
</div>

Les utilisateurs de X ont, de façon organique, introduit de nouveaux schémas de communication désormais fondamentaux au sein du réseau X. Un exemple marquant est le hashtag, aujourd’hui presque universel sur l’ensemble des réseaux sociaux. Les hashtags ont été introduits comme un moyen d’organiser les conversations et les sujets. Sur un réseau où des centaines de millions de messages sont publiés chaque jour, disposer d’outils pour trouver des Posts pertinents est essentiel, et les hashtags sont devenus un mécanisme clé. Peu après l’essor des hashtags, X leur a conféré un statut et un support officiels. Lorsqu’ils sont devenus un *« objet de première classe »*, cela a entraîné plusieurs conséquences. D’une part, les hashtags sont devenus cliquables/recherchables dans l’interface utilisateur de X.com. D’autre part, ils ont rejoint la famille des *entities* de X, aux côtés des @mentions, des médias joints, des symboles boursiers et des liens partagés. Ces entités sont commodément encodées dans un tableau JSON pré-analysé, ce qui facilite leur traitement, leur indexation et leur stockage par les développeurs.

Les Retweets sont un autre exemple de conventions initiées par les utilisateurs devenant des objets officiels. Le Retweet a émergé comme un moyen de « transférer » du contenu à d’autres. Au départ, il s’agissait d’un processus manuel consistant à copier/coller un Post et à le faire précéder du motif « RT @ ». Ce processus a ensuite été automatisé via un nouveau bouton Retweet, accompagné de nouvelles métadonnées JSON. Le Retweet « officiel » était né. D’autres exemples incluent les « mentions », le partage de médias et de liens web, ainsi que le partage d’une localisation avec votre Post. Chacune de ces pratiques d’usage a donné lieu à de nouvelles fonctionnalités de l’interface utilisateur sur [x.com](https://x.com/), à de nouveaux JSON de support, et donc à de nouvelles façons d’effectuer des correspondances sur des Posts. L’ensemble de ces attributs fondamentaux des Posts ont abouti à des PowerTrack Operators permettant de les cibler.

<div id="post-metadata-mutability-updates-and-currency">
  #### Métadonnées des Posts, mutabilité, mises à jour et actualité
</div>

Bien que les messages d’un Post puissent comporter jusqu’à un nombre fixe de caractères, la description JSON d’un Post comprend plus de 100 attributs. Des attributs tels que qui a publié, à quel moment, s’il s’agit d’un Post original ou d’un Retweet, ainsi qu’un tableau d’objets de premier niveau tels que les hashtags, les mentions et les liens partagés. Pour le compte qui a publié, il existe un objet User (ou Actor) avec une variété d’attributs qui fournissent le profil de l’utilisateur et d’autres métadonnées de compte. Les profils incluent une brève biographie, un lieu de résidence (texte libre), la langue préférée et un lien de site web facultatif.

Certaines métadonnées de compte ne changent jamais (par exemple l’ID utilisateur numérique et la date de création), d’autres évoluent lentement au fil du temps, tandis que certains attributs changent plus fréquemment. Les personnes changent d’emploi et déménagent. Les entreprises mettent à jour leurs informations. Lorsque vous collectez des Posts historiques, il est important de comprendre que certaines métadonnées sont *telles qu’elles étaient au moment de la publication*, et d’autres *telles qu’elles sont au moment où la requête est soumise*. 

Avec toutes les API historiques, la description de profil de l’utilisateur, le nom d’affichage et les attributs « home » du profil sont mis à jour avec les valeurs en vigueur au moment de la requête.

<div id="native-media">
  #### Médias « natifs »
</div>

X.com et les applications mobiles X permettent d’ajouter des photos et des vidéos à une publication en cliquant sur un bouton et en parcourant vos galeries. Désormais intégrées comme actions de premier plan, les vidéos et photos partagées de cette façon sont appelées des médias « natifs ».

De nombreux opérateurs de requête fonctionnent avec ces ressources « natives », notamment `has:videos`, `has:images` et `has:media`. Ceux-ci correspondent uniquement au contenu média partagé via les fonctionnalités de X. Pour faire correspondre d’autres médias hébergés hors de la plateforme X, vous devrez utiliser des opérateurs qui ciblent les métadonnées d’URL.

Avant d’entrer dans les détails des produits Historical PowerTrack et Full-Archive Search, faisons un tour de l’évolution de X, en tant que produit et plateforme, au fil du temps.

**Conseils de filtrage**

Être familier de la chronologie X indiquant quand et comment de nouvelles fonctionnalités ont été ajoutées peut vous aider à créer des requêtes plus efficaces. Ici, une requête désigne un *filtre* ou une *règle* appliqué par les API historiques X à l’archive des Posts, en utilisant des Operators PowerTrack pour faire correspondre le JSON des Posts. Par exemple, l’Operator `lang:` sert à faire correspondre les Posts dans une langue donnée. X fournit un service de classification des langues (plus de 50 langues prises en charge) et les API X exposent ces métadonnées dans le JSON généré pour chaque Post. Ainsi, si un Post est rédigé en espagnol, l’attribut JSON « lang » est défini sur « es ». Donc, si vous créez un filtre avec la clause `lang:es`, il ne correspondra qu’aux messages de Post classés comme espagnols.

Les informations de chronologie peuvent également aider à mieux interpréter les données de Posts reçues. Supposons que vous étudiiez le partage de contenu sur les Jeux olympiques d’été de 2008 et 2012. Si vous appliquiez uniquement l’Operator `is:retweet` pour faire correspondre les Retweets, aucune donnée ne correspondrait en 2008. En revanche, pour 2012, il y aurait probablement des millions de Retweets. Vous pourriez alors conclure à tort qu’en 2008 les Retweets n’étaient pas une pratique des utilisateurs, ou que personne n’a simplement retweeté à propos de ces Jeux olympiques. Étant donné que les Retweets sont devenus un objet de première classe en 2009, vous devez ajouter une clause de règle « RT @ » pour aider à les identifier en 2008.

Les Retweets et la classification linguistique des Posts sont deux exemples d’attributs de Post avec une longue histoire et de nombreux détails produits. Ci-dessous, nous détaillerons davantage ceux-ci ainsi que d’autres catégories d’attributs importantes pour faire correspondre et comprendre les données X.

<div id="recognizing-false-negatives">
  #### Reconnaître les faux négatifs
</div>

Lorsqu’il s’agit d’écrire des filtres, un point important à retenir est que tous les opérateurs basés sur des métadonnées ont des dates d’« introduction ». Si vous créez un filtre avec un opérateur qui agit sur des métadonnées apparues après la publication du Post, vous obtiendrez un faux négatif. Par exemple, disons que vous vous intéressez à tous les Posts qui mentionnent “snow” et partagent une vidéo. Si vous créez une règle avec l’opérateur `has:videos`, qui correspond aux Posts avec des vidéos natives, cette clause ne correspondra à aucun Post antérieur à 2015.

Cependant, le partage de vidéos était courant sur X bien avant 2015. Avant cela, les utilisateurs partageaient des liens vers des vidéos hébergées ailleurs, mais en 2015, X a intégré de nouvelles fonctionnalités de partage de vidéos directement dans la plateforme. Pour retrouver ces anciens Posts, vous incluriez une clause de règle telle que `url:"youtube.com"`.

Notez qu’avec les API de recherche, il existe quelques exemples de métadonnées ayant été « rétroremplies » lors de la reconstruction de l’index. Un bon exemple concerne les $cashtags, devenus largement utilisés pour discuter des symboles boursiers en 2009. Après l’introduction de l’opérateur $cashtag en 2015, l’index de recherche a été reconstruit et, au cours du processus, l’entité symbole a été extraite de tous les corps de Post, y compris dès 2006, lorsque le caractère `$` était principalement utilisé comme argot : “I hope it $now$ $oon!”.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### Identifier et filtrer selon les attributs de Post importants pour votre cas d’usage
</div>

Certaines métadonnées, telles que les identifiants numériques de compte X, existent depuis le premier jour (et constituent un exemple de métadonnées de compte qui ne changent jamais). D’autres métadonnées n’ont été introduites que bien après le lancement de X en 2006. Parmi les nouvelles métadonnées introduites figurent celles liées aux Retweets, les localisations des Posts, les titres et descriptions d’URL, ainsi que les médias « natifs ». Vous trouverez ci-dessous certains des types d’attributs de Post les plus courants qui ont été profondément affectés par ces mises à jour de la plateforme X.

Le comportement de filtrage/rapprochement dépend, dans la plupart des cas, de l’API de Post historique utilisée. Pour vous aider à déterminer quel produit convient le mieux à votre recherche et à votre cas d’usage, les détails d’attributs fournis ci-dessous incluent des informations produit de haut niveau.

<div id="x-profiles">
  #### Profils X
</div>

Étant donné qu’au cœur de X se trouve un canal de communication mondial en temps réel, les recherches utilisant des données de Post mettent généralement l’accent sur qui s’exprime. Il est souvent utile de savoir quel endroit un utilisateur X considère comme son domicile. Savoir que la bio d’un compte mentionne des centres d’intérêt et des hobbies peut également vous orienter vers des Posts pertinents. Il est très courant de vouloir écouter les Posts provenant de comptes d’intérêt. Les attributs de profil sont essentiels pour tous ces cas d’usage.

Chaque compte sur X possède un profil qui inclut des métadonnées telles que le @handle X, le nom d’affichage, une courte bio, la localisation (texte libre saisi par l’utilisateur), le nombre d’abonnés, et bien d’autres. Certains attributs ne changent jamais, comme l’identifiant utilisateur numérique et la date de création du compte. D’autres évoluent généralement au jour le jour, de semaine en semaine ou de mois en mois, comme le nombre de Posts publiés et le nombre de comptes suivis et d’abonnés. D’autres attributs de compte peuvent également changer à tout moment, mais ont tendance à évoluer moins fréquemment : nom d’affichage, localisation et bio.

La charge utile JSON de chaque Post inclut des métadonnées de *profil de compte* pour l’auteur du Post. S’il s’agit d’un Retweet, elle inclut également les métadonnées de profil du compte qui a publié le Post original.

La mutabilité des métadonnées de profil d’un Post dépend entièrement du produit historique utilisé. Les API Search servent des Posts historiques avec les paramètres de profil *tels qu’ils sont au moment de la récupération*. Pour Historical PowerTrack, le profil est *tel qu’il était au moment où le Post a été publié*, à l’exception des données antérieures à 2011. Pour les Posts antérieurs à 2011, les métadonnées de profil reflètent le profil tel qu’il était en septembre 2011.

<div id="original-post-and-retweets">
  #### Publication originale et Retweets
</div>

Les Retweets sont un autre exemple de conventions initiées par les utilisateurs devenant des objets officiels. Le retweet est apparu comme un moyen de « faire suivre » du contenu à d’autres. Il a commencé comme un processus manuel consistant à copier-coller une Publication et à la faire précéder du motif « RT @ ». Ce processus a ensuite été automatisé via un nouveau bouton Retweet, accompagné de nouvelles métadonnées JSON. Le Retweet « officiel » est né et l’action de retweeter est devenue un événement de Publication de premier ordre. Avec le nouveau bouton Retweet, de nouvelles métadonnées ont été introduites, telles que la charge utile complète de la Publication originale.

Le fait qu’une Publication soit originale ou partagée constitue un critère de filtrage courant. Dans certains cas, seul le contenu original est nécessaire. Dans d’autres, l’engagement sur la Publication est primordial, et les Retweets sont donc essentiels. L’opérateur PowerTrack `is:retweet` permet d’inclure ou d’exclure les Retweets. Si vous extrayez des données antérieures à août 2009, il vous faut deux stratégies pour identifier (ou non) les Retweets. Avant août 2009, il faut vérifier le message de la Publication lui-même, en utilisant une correspondance exacte d’expression, afin de repérer le motif « RT @ ». Pour les périodes postérieures à août 2009, l’opérateur `is:retweet` est disponible.

<div id="post-language-classifications">
  #### Classifications de langue des Posts
</div>

La langue dans laquelle un Post est rédigé est un intérêt fréquent. La langue du Post peut aider à déduire l’emplacement d’un Post et, souvent, une langue spécifique suffit pour l’analyse ou l’affichage. (Les profils X disposent également d’un paramètre de langue préférée.)

Pour le filtrage basé sur la classification linguistique d’un Post, les produits historiques de X ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) et Historical PowerTrack) diffèrent sensiblement. Lors de la constitution de l’archive Search, tous les posts ont été rétrospectivement complétés avec la classification linguistique de X. Par conséquent, l’opérateur `lang:` est disponible pour l’ensemble de l’archive des posts. Avec Historical PowerTrack, les métadonnées de classification linguistique de X sont disponibles dans l’archive à partir du 26 mars 2013. 

<div id="geo-referencing-posts">
  #### Géoréférencement des Posts
</div>

Pouvoir indiquer où un Post a été publié (c’est-à-dire le géoréférencer) est important pour de nombreux cas d’usage. Il existe trois méthodes principales pour géoréférencer des Posts :

* Références géographiques dans le contenu d’un Post
* Posts géolocalisés par l’utilisateur
* Lieu « domicile » du profil de compte défini par un utilisateur

<div id="geographical-references-in-a-post-message">
  ##### Références géographiques dans un message Post
</div>

Le repérage par références géographiques dans le message Post, bien que souvent la méthode la plus difficile puisqu’il repose sur des connaissances locales, est possible sur l’ensemble des archives de Posts. Voici un exemple d’association géoréférencée de 2006 pour la région de San Francisco, basé sur un filtre « golden gate » :

https://x.com/biz/statuses/28311

<div id="posts-geo-tagged-by-the-user">
  ##### Publications géolocalisées par l’utilisateur
</div>

En novembre 2009, X a lancé son API de géomarquage des publications, qui permettait d’associer aux publications une localisation précise. En juin 2010, X a lancé X Places, qui représente une zone géographique à l’échelle d’un lieu, d’un quartier ou d’une ville. Environ 1 à 2 % des publications sont géolocalisées via l’une ou l’autre de ces méthodes.

L’historique de géolocalisation disponible dépend de l’API historique que vous utilisez. Avec les API Search, la possibilité de commencer à faire correspondre des publications à l’aide de certains opérateurs géographiques a débuté en mars 2010, et avec d’autres en février 2015. Si vous utilisez Historical PowerTrack, la géoréférencement commence le 1er septembre 2011. Lors de la création de l’archive Historical PowerTrack, toute géolocalisation antérieure à cette date n’a pas été incluse.

<div id="account-profile-home-location-set-by-a-user">
  ##### Lieu « domicile » du profil de compte défini par l’utilisateur
</div>

Tous les utilisateurs de X peuvent renseigner le Lieu de profil, indiquant leur domicile. Des millions d’utilisateurs fournissent cette information, ce qui accroît considérablement la quantité de données géographiques dans le X Firehose. Ces métadonnées de localisation sont une chaîne libre, non normalisée et générée par l’utilisateur. Environ 30 % des comptes disposent de métadonnées Profile Geo pouvant être résolues au niveau du pays.

Comme pour la géolocalisation des Posts, les méthodes de correspondance et les périodes disponibles dépendent de l’API historique que vous utilisez. Historical PowerTrack permet d’effectuer vos propres correspondances personnalisées sur ces chaînes libres. Pour faciliter ce processus, X propose également un [Profile Geo Enrichment](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) qui réalise le géocodage lorsque c’est possible, en fournissant des métadonnées normalisées et des opérateurs correspondants. Les opérateurs Profile Geo sont disponibles dans Historical PowerTrack comme dans les Search APIs. Avec Historical PowerTrack, ces métadonnées Profile Geo sont disponibles depuis juin 2014. Avec les Search APIs, ces métadonnées sont disponibles depuis février 2015.

<div id="shared-links-and-media">
  #### Liens partagés et médias
</div>

Le partage de liens vers des pages web, de photos et de vidéos a toujours été un cas d’usage essentiel de X. Au début, toutes ces actions impliquaient d’inclure une URL dans le message de la Publication elle-même. En 2011, X a intégré le partage de photos directement dans son interface utilisateur. En 2016, les vidéos natives ont été ajoutées.

Compte tenu de cet historique, il existe une variété d’opérateurs de filtrage utilisés pour cibler ce contenu. Un ensemble d’opérateurs permet de déterminer si les Publications contiennent des liens partagés, des photos et des vidéos. De plus, comme la plupart des URL partagées sur X sont raccourcies afin d’utiliser moins de caractères d’une Publication (par exemple générées par un service tel que bitly ou tinyurl), X fournit des enrichissements de données qui génèrent une URL complète et développée sur laquelle il est possible de faire correspondre. Par exemple, si vous souhaitez cibler des Publications incluant des liens traitant de X et des systèmes d’alerte précoce, un filtre faisant référence à « severe weather communication » correspondrait à une Publication contenant cette URL [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4).

En mars 2012, l’[enrichissement d’URL développée](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) a été introduit. Avant cette date, les charges utiles des Publications incluaient uniquement l’URL telle que fournie par l’utilisateur. Ainsi, si l’utilisateur incluait une URL raccourcie, il pouvait être difficile de faire correspondre les URL (développées) d’intérêt. Avec Historical PowerTrack et les Search APIs, ces métadonnées sont disponibles à partir de mars 2012.

En juillet 2016, l’enrichissement d’URL améliorée a été introduit. Cette version améliorée fournit le titre HTML et la description d’un site web dans la charge utile de la Publication, ainsi que des opérateurs pour établir des correspondances sur ces éléments. Avec Historical PowerTrack, ces métadonnées deviennent disponibles en juillet 2016. Avec les Search APIs, ces métadonnées commencent à apparaître en décembre 2014.

En septembre 2016, X a introduit les « pièces jointes natives » où un lien partagé final n’est pas décompté des 140 caractères d’une Publication. Les deux enrichissements d’URL s’appliquent toujours à ces liens partagés.

Pour d’autres détails spécifiques au produit concernant le filtrage des URL, consultez les articles correspondants pour plus d’informations.
