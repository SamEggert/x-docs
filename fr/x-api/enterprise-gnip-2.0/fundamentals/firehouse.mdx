Cet outil vous permet de téléverser de vastes ensembles d’ID de publications ou d’utilisateurs afin d’obtenir leur statut de conformité et de déterminer quelles données nécessitent une action pour mettre vos jeux de données en conformité.

<div id="overview">
  ## Aperçu
</div>

`Enterprise`

L’une des valeurs fondamentales de X est de défendre et de respecter la voix des utilisateurs. Cela inclut le respect de leurs attentes et de leur intention lorsqu’ils suppriment, modifient ou éditent le contenu qu’ils choisissent de partager sur X. Nous estimons que cela est crucial pour la santé à long terme de l’une des plus grandes plateformes publiques d’information en temps réel au monde. X met le contrôle entre les mains de ses utilisateurs, leur donnant la possibilité de maîtriser leur propre expérience sur X. Nous pensons que les clients professionnels qui reçoivent des données X ont la responsabilité de respecter les attentes et l’intention des utilisateurs finaux.

Pour plus d’informations sur les types d’événements de conformité possibles sur la plateforme X, consultez notre article [Respecter l’intention des utilisateurs sur X](/fr/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Tout développeur ou entreprise consommant des données X via une API a l’obligation de déployer tous les efforts raisonnables pour respecter les modifications apportées au contenu des utilisateurs. Cette obligation s’étend aux événements utilisateur tels que les suppressions, modifications et changements d’options de partage (par exemple, contenu devenu protégé ou soumis à des restrictions). Cela concerne également les cas où les utilisateurs éditent leurs publications. Veuillez vous référer au libellé spécifique de la [Developer Policy](https://developer.x.com/en/developer-terms/policy) et/ou de votre X Data Agreement pour comprendre comment cette obligation affecte votre utilisation des données X.

X propose les solutions suivantes, qui fournissent des informations sur ces événements de conformité utilisateur et indiquent si une publication ou un compte spécifique est publiquement accessible ou non. Un bref aperçu des solutions et de leurs modèles d’intégration généraux est présenté ci-dessous :

<div id="get-statuseslookup-and-get-userslookup">
  #### GET statuses/lookup et GET users/lookup
</div>

* Format : API REST. Voir [GET statuses/lookup](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) et [GET users/lookup](https://developer.x.com/en/docs/x-api/v1/accounts-and-users/follow-search-get-users/api-reference/get-users-lookup).

* Ces endpoints renvoient toujours la dernière version de toute édition de Post. Tous les objets Post décrivant des Posts créés après l’introduction de la fonctionnalité d’édition incluront les métadonnées d’édition. Cela est vrai même pour les Posts qui n’ont pas été modifiés.

* Pour tous les Posts, les requêtes effectuées plus de 30 minutes après leur création représenteront l’état final des Posts.

* Fournissent des informations de disponibilité pour des Posts ou des Users spécifiques, tels que définis par l’appelant dans la requête API.

* Peuvent être utilisés pour des vérifications ponctuelles ad hoc de l’état de disponibilité actuel d’un groupe spécifique de Posts/Users.

* Idéal pour les clients qui ont besoin d’un moyen de vérifier l’état actuel d’un Post ou d’un User spécifique à un instant donné.

* Ces APIs offrent un mécanisme utile qui peut être utilisé par les clients qui doivent vérifier la disponibilité d’un élément de contenu, par exemple lorsque :
  1. Afficher des Posts
  2. Interagir de manière 1:1 avec un ou plusieurs Post(s) ou User(s)
  3. Distribuer du contenu X à un tiers via un téléchargement de fichier autorisé
  4. Stocker des Posts pendant de longues périodes

<div id="compliance-firehose-enterprise-only">
  #### Compliance Firehose (entreprise uniquement)
</div>

* Format : API de streaming. Voir : [Compliance Firehose](/fr/x-api/enterprise-gnip-2.0/fundamentals/firehouse#api-reference).
* Fournit un flux en temps réel des [activités de conformité](/fr/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) sur X. Ces activités incluent notamment l’édition de publications.
* Peut être utilisée pour maintenir l’état de conformité d’un ensemble de données stockées à mesure que de nouveaux événements de conformité surviennent sur la plateforme.
* Idéal pour les clients qui consomment et stockent de grandes quantités de données X sur de longues périodes.

<div id="compliance-best-practices">
  ### **Meilleures pratiques en matière de conformité**
</div>

<div id="recommendations-best-practices">
  #### **Recommandations et bonnes pratiques**
</div>

* **Concevoir des schémas de stockage qui conservent l’ID numérique du Post et l’ID numérique de l’utilisateur** : Les messages relatifs aux utilisateurs nécessitent d’agir sur tous les Posts de cet utilisateur. Par conséquent, puisque tous les messages de conformité sont transmis uniquement par ID numérique, il est important de concevoir des schémas qui maintiennent la relation entre le Post et l’utilisateur sur la base des IDs numériques. Les consommateurs de données devront surveiller les événements de conformité à la fois par ID de Post et par ID d’utilisateur, et être en mesure de mettre à jour correctement le datastore local.

* **Concevoir des schémas qui couvrent tous les statuts de conformité** : Selon la manière dont les activités de conformité seront gérées dans différentes applications, il peut être nécessaire d’ajouter d’autres métadonnées au datastore. Par exemple, les consommateurs de données peuvent décider d’ajouter des métadonnées à une base de données existante pour faciliter la restriction de l’affichage de contenu dans les pays concernés par un message status&#95;withheld.

* **Gestion des suppressions de Retweets** : Les Retweets sont un type particulier de Post où le message original est imbriqué dans un objet au sein du Retweet. Dans ce cas, deux IDs de Post sont référencés dans un Retweet : l’ID du Retweet et l’ID du message original (inclus dans l’objet imbriqué). Lorsqu’un message original est supprimé, un message de suppression de Post est émis pour l’ID original. Les événements de suppression de Post déclenchent généralement des événements de suppression pour tous les Retweets. Cependant, dans certains cas, tous ne sont pas envoyés et les systèmes clients doivent tolérer des suppressions de Retweets incomplètes. La suppression de l’ID original devrait suffire à supprimer tous les Retweets ultérieurs. Bonne pratique : référencer l’ID du Post original lors du stockage des Retweets, et supprimer tous les Retweets référencés lors de la réception d’événements de suppression de Post.

Les types possibles d’événements de conformité incluent des événements Post (ou « status ») et des événements User, dont plusieurs types sont décrits ci-dessous.  

Veuillez noter :

* En savoir plus sur les statuts User [ici](https://support.x.com/articles/14016) et sur notre politique développeur concernant les Posts supprimés [ici](https://dev.x.com/overview/terms/policy#3.Update_Respect_Users_Control_and_Privacy).
* Le Compliance Firehose a été mis à jour pour fournir des événements « tweet&#95;edit ».
* Plusieurs événements de suppression, de protection et de suspension de User ne sont pas nécessairement permanents et peuvent alterner entre différents états indéfiniment. Il s’agit de : user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect, user&#95;suspend et user&#95;unsuspend.
* Les user&#95;delete sont suivis de status&#95;delete 30 jours plus tard uniquement si l’utilisateur n’a pas choisi de user&#95;undelete son compte. Il est possible qu’un user&#95;delete soit annulé par l’utilisateur et que les suppressions de tous ses Posts 30 jours plus tard n’aient pas lieu.
* user&#95;suspend est une action qui reste effective, sauf si l’utilisateur fait l’objet d’un événement user&#95;unsuspend. Ces événements ne sont soumis à aucun changement sur une période de 30 jours.

Reportez-vous à la colonne « Recommended Action » pour comprendre comment traiter chaque type d’événement afin de respecter la confidentialité et l’intention de l’utilisateur final.

| Original Message Type | Object | Permanent (Yes/No) | Recommended Action |
| :--- | :--- | :--- | :--- |
| delete | Status | Yes | Supprimer le Post associé. |

| user&#95;undelete | User | No  | Tous les Posts de l’utilisateur associé peuvent être affichés à nouveau et traités comme publics. |
| user&#95;protect | User | No  | Masquer ou supprimer tous les Posts de l’utilisateur associé. |
| user&#95;unprotect | User | No  | Tous les Posts de l’utilisateur associé peuvent être affichés à nouveau et traités comme publics. |

<div id="integrating-compliance-firehose">
  ### intégration de Compliance Firehose
</div>

La Compliance Firehose est une API de diffusion en continu en temps réel qui fournit les événements de conformité se produisant sur la plateforme X. Pour comprendre les événements de conformité et la façon dont ils sont générés sur X, veuillez consulter notre article [Respecter l’intention des utilisateurs sur X](/fr/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Il est important de noter que les événements Post et User sont délivrés séparément et que chacun doit être traité indépendamment (c.-à-d. la suppression d’un Post n’implique pas un événement User, et inversement). Plusieurs événements User ne sont pas nécessairement permanents et peuvent alterner entre différents états indéfiniment. Ceux-ci incluent : user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect, user&#95;suspend et user&#95;unsuspend.

Les user&#95;deletes sont suivis de status&#95;deletes 30 jours plus tard uniquement si l’utilisateur n’a pas choisi d’effectuer un user&#95;undelete de son compte. Il est donc possible qu’un user&#95;delete soit annulé par l’utilisateur et que les suppressions de tous ses Posts 30 jours plus tard n’aient pas lieu.

L’état user&#95;suspend reste en vigueur sauf si l’utilisateur fait l’objet d’un événement user&#95;unsuspend. Ces événements ne sont soumis à aucun changement sur une période de 30 jours.

Il n’est jamais approprié d’afficher directement les événements de conformité aux utilisateurs de votre logiciel ni de les intégrer à vos produits ou expériences client. Ils sont destinés uniquement à maintenir la conformité et à respecter les actions des utilisateurs de X.

<div id="integrating-with-the-compliance-firehose">
  #### **Intégration du Compliance Firehose**
</div>

Pour intégrer le Compliance Firehose à votre système, vous devrez créer une intégration capable de faire ce qui suit :

1. Établir une connexion en streaming à chaque partition d’API de streaming du Compliance Firehose
2. Gérer des volumes de données élevés – découpler l’ingestion du flux du traitement en aval à l’aide de processus asynchrones
3. Se reconnecter automatiquement aux partitions du flux en cas de déconnexion, quelle qu’en soit la raison
4. Traiter les événements de conformité pertinents pour les données Post et User que vous avez stockées, conformément aux recommandations présentées ci‑dessus

<div id="honoring-user-intent-on-twitter">
  ### Respecter l’intention des utilisateurs sur X
</div>

Nous estimons que le respect de la vie privée et de l’intention des utilisateurs de X est essentiel à la pérennité de l’une des plus grandes plateformes publiques d’information en temps réel au monde. X place les contrôles de confidentialité entre les mains de ses utilisateurs, leur donnant la possibilité de maîtriser leur propre expérience sur X. En tant qu’utilisateurs professionnels des données de X, nous avons la responsabilité collective de respecter la vie privée et les actions des utilisateurs finaux afin de préserver cet environnement de confiance et de respect.

Divers événements peuvent survenir concernant les publications et les comptes utilisateurs et influencer la manière dont ils sont affichés sur la plateforme. Les actions qui affectent la confidentialité et l’intention sont définies aux niveaux Publication (Post) et Utilisateur. Ces actions incluent :

<div id="user">
  #### Utilisateur
</div>

| Action | Description |
| :--- | :--- |
| Protéger le compte | Un utilisateur X peut protéger ou retirer la protection de son compte à tout moment. Les comptes protégés exigent l’approbation manuelle par l’utilisateur de chaque personne autorisée à voir les Posts de son compte.   <br />Pour plus d’informations, consultez [À propos des Posts publics et protégés](https://support.x.com/articles/14016-about-public-and-protected-tweets). |
| Supprimer le compte | Un utilisateur X peut décider de supprimer son compte et tous les messages de statut associés à tout moment. X conserve les informations du compte pendant 30 jours après la suppression au cas où l’utilisateur déciderait d’annuler la suppression et de réactiver son compte. |
| Nettoyer la géolocalisation | Un utilisateur X peut supprimer à tout moment toutes les données de localisation de ses Posts passés. Cela est appelé « scrub geo ». |
| Suspendre le compte | X se réserve le droit de suspendre les comptes qui enfreignent les Règles de X ou si un compte est soupçonné d’avoir été piraté ou compromis. Les suspensions de compte ne peuvent être levées (unsuspend) que par X. |
| Retenir le compte | X se réserve le droit de restreindre de manière réactive l’accès à certains contenus dans un pays spécifique, de temps à autre. Un compte retenu ne peut être libéré de cette retenue que par X.   <br />Pour plus d’informations, consultez [Contenu restreint par pays](https://support.x.com/articles/20169222-country-withheld-content). |

| Action | Description |
| :--- | :--- |
| Supprimer le statut | Un utilisateur X peut supprimer un statut à tout moment. Les statuts supprimés ne peuvent pas être rétablis et sont définitivement supprimés. |
| Retenir le statut | X se réserve le droit de restreindre, de manière réactive, l’accès à certains contenus dans un pays spécifique, de temps à autre. Un statut retenu ne peut être rendu à nouveau accessible que par X.  <br />Pour plus d’informations, consultez [Contenu retenu par pays](https://support.x.com/articles/20169222-country-withheld-content). |

<div id="keeping-track-of-user-and-status-changes">
  #### Suivi des changements d’utilisateur et de statut
</div>

L’état d’un utilisateur ou d’un statut peut changer à tout moment à la suite de l’une des actions ci-dessus, ce qui affecte la manière dont les consommateurs des données X doivent traiter la disponibilité et la confidentialité de tout contenu associé. Lorsque ces actions se produisent, un message de conformité correspondant est envoyé pour indiquer que l’état d’un utilisateur ou d’un statut a changé.

<div id="get-compliancestream">
  #### GET /compliance/:stream
</div>

Établit une connexion persistante au flux de données Compliance firehose, par lequel les événements de conformité seront livrés.

|     |     |
| :--- | :--- |
| **Request Method** | HTTP GET |
| **Connection Type** | Keep-Alive |
| **URL** | Disponible sur la page d’aide de l’API du flux dans votre tableau de bord et suit la structure suivante :  <br />  <br /><br />[https://gnip-stream.x.com/stream/compliance/accounts/:account&#95;name/publishers/twitter/:stream&#95;label.json?partition=1](https://gnip-stream.x.com/stream/compliance/accounts/:account_name/publishers/twitter/:stream_label.json?partition=1)<br /><br />**Remarque :** Le paramètre « partition » est requis. Vous devrez vous connecter aux 8 partitions, chacune contenant 12,5 % du volume total, pour consommer l’intégralité du flux. |
| **Compression** | Gzip. Pour vous connecter au flux avec la compression Gzip, envoyez simplement un en-tête Accept-Encoding dans la requête de connexion. L’en-tête doit être le suivant :  <br />  <br />Accept-Encoding: gzip |
| **Character Encoding** | UTF-8 |
| **Response Format** | JSON. L’en-tête de votre requête doit spécifier le format JSON pour la réponse. |
| **Rate Limit** | 10 requêtes par 60 secondes. |
| **Read Timeout** | Configurez un délai d’attente de lecture côté client et assurez-vous qu’il est supérieur à 30 secondes. |
| **Support for Tweet edits** | Toute modification d’un Tweet déclenche un événement de conformité « tweet&#95;edit ». Consultez la documentation [Compliance Data Objects](/fr/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) pour plus de détails. |

**Exemple de requête Curl**

La requête d’exemple suivante est effectuée à l’aide de cURL en ligne de commande :

```bash
curl --compressed -v -uexample@customer.com "https://gnip-stream.x.com/stream/compliance/accounts/:account_name/publishers/twitter/:stream_label.json?partition=1"
```

*Remarque :* la requête ci-dessus ne se connecte qu’à `partition=1` du Compliance firehose — vous devrez vous connecter aux 8 partitions pour consommer l’intégralité de ce flux.

<div id="response-codes">
  #### Codes de réponse
</div>

Les réponses suivantes peuvent être renvoyées par l’API pour ces requêtes. La plupart des codes d’erreur sont renvoyés avec une chaîne contenant des détails supplémentaires dans le corps de la réponse. Pour les réponses autres que 200, les clients doivent tenter de se reconnecter.

| Statut | Texte | Définition |
| :--- | :--- | :--- |
| 200 | Succès | La connexion a été ouverte avec succès et de nouvelles activités seront envoyées au fur et à mesure de leur arrivée. |
| 401 | Non autorisé | L’authentification HTTP a échoué en raison d’identifiants invalides. Connectez-vous à console.gnip.com avec vos identifiants pour vérifier que vous les utilisez correctement dans votre requête. |
| 406 | Non acceptable | En général, cela se produit lorsque votre client n’inclut pas correctement les en-têtes pour accepter l’encodage gzip du flux, mais cela peut aussi se produire dans d’autres circonstances. Le corps contiendra un message JSON similaire à « Cette connexion requiert une compression. Pour activer la compression, envoyez un en-tête “Accept-Encoding: gzip” dans votre requête et préparez-vous à décompresser le flux au fur et à mesure de sa lecture côté client. » |
| 429 | Limite de débit atteinte | Votre application a dépassé la limite de demandes de connexion. |
| 503 | Service indisponible | Problème côté serveur X. Reconnectez-vous en utilisant une stratégie de réessais exponentiels. Si aucun avis concernant ce problème n’a été publié sur la [X API Status Page](https://api.twitterstat.us/), contactez l’assistance. |

<div id="other-recommendations-best-practices">
  #### Autres recommandations et bonnes pratiques
</div>

* **Concevez des schémas de stockage de données qui conservent l’ID de Tweet et l’ID d’utilisateur numériques** : Les messages liés aux utilisateurs exigent d’agir sur tous les Tweets de cet utilisateur. Comme tous les messages de conformité ne sont fournis que via des IDs numériques, il est important de concevoir des schémas de stockage qui maintiennent la relation entre le Tweet et l’utilisateur sur la base de ces IDs. Les consommateurs de données devront suivre les événements de conformité à la fois par ID de Tweet et par ID d’utilisateur, et être en mesure de mettre à jour le magasin de données local en conséquence.

* **Concevez des schémas qui couvrent tous les statuts de conformité** : Selon la manière dont les activités de conformité seront gérées dans différentes applications, il peut être nécessaire d’ajouter d’autres métadonnées au magasin de données. Par exemple, les consommateurs de données peuvent décider d’ajouter des métadonnées à une base de données existante pour faciliter la restriction de l’affichage de contenu dans les pays concernés par un message status&#95;withheld.

* **Gestion des suppressions de Retweets** : Les Retweets sont un type particulier de Tweet où le message d’origine est imbriqué dans un objet à l’intérieur du Retweet. Dans ce cas, deux IDs de Tweet sont référencés dans un Retweet — l’ID du Retweet et l’ID du message d’origine (inclus dans l’objet imbriqué). Lorsqu’un message d’origine est supprimé, un message de suppression de Tweet est émis pour l’ID d’origine. Des messages de suppression ultérieurs NE sont PAS émis pour tous les Retweets. La suppression de l’ID d’origine suffit à supprimer tous les Retweets ultérieurs.
