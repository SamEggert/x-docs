<div id="authentication">
  #### Authentification
</div>

Tous les endpoints de l’API X v2 exigent que vous authentifiiez vos requêtes avec un jeu d’identifiants, également appelé clés et jetons. Vous pouvez utiliser OAuth 1.0a User Context, OAuth 2.0 Authorization Code with PKCE ou App only pour authentifier vos requêtes vers les endpoints de **lookup** des Lists. Cependant, vous devez vous authentifier avec OAuth 1.0a User Context ou OAuth 2.0 pour les endpoints **manage** des Lists.

[OAuth 1.0a User Context](/fr/resources/fundamentals/authentication), ce qui signifie que vous devez utiliser un jeu d’API Keys et de user Access Tokens pour effectuer une requête avec succès. Les access tokens doivent être associés à l’utilisateur pour le compte duquel vous effectuez la requête. Si vous souhaitez générer un jeu d’Access Tokens pour un autre utilisateur, il doit autoriser votre App via le [flux OAuth à 3 étapes](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Veuillez noter qu’OAuth 1.0a peut être difficile à utiliser. Si vous n’êtes pas familier avec cette méthode d’authentification, nous vous recommandons d’utiliser une [bibliothèque](/fr/x-api/tools-and-libraries/overview), un outil comme Postman, ou d’utiliser OAuth 2.0 ou App only pour authentifier vos requêtes.

[OAuth 2.0 Authorization Code with PKCE](/fr/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) offre un meilleur contrôle sur la portée d’une application et permet des flux d’autorisation sur plusieurs appareils. OAuth 2.0 vous permet de choisir des scopes précis et granulaires qui vous accordent des autorisations spécifiques au nom d’un utilisateur.

Pour activer OAuth 2.0 dans votre App, vous devez l’activer dans les paramètres d’authentification de votre App, situés dans la section App settings du developer portal.

[App only](/fr/resources/fundamentals/authentication#oauth-2-0) requiert simplement d’inclure un [App only Access Token](/fr/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) dans votre requête. Vous pouvez générer un App only Access Token directement dans une developer App, ou en générer un via l’endpoint [POST oauth2/token](/fr/resources/fundamentals/authentication#post-oauth2-token).

[App only](/fr/resources/fundamentals/authentication#oauth-2-0) requiert simplement d’inclure un [App only Access Token](/fr/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) dans votre requête. Vous pouvez générer un App only Access Token directement dans une developer App, ou en générer un via l’endpoint [POST oauth2/token](/fr/resources/fundamentals/authentication#post-oauth2-token).

<div id="developer-portal-projects-and-developer-apps">
  #### Portail développeur, projets et applications développeur
</div>

Pour obtenir un jeu d’informations d’identification compatible avec les endpoints de l’API X v2, vous devez [créer un compte développeur](https://developer.x.com/en/portal/petition/essential/basic-info), configurer un [projet](/fr/resources/fundamentals/projects) dans ce compte, puis créer une [application développeur](/fr/resources/fundamentals/developer-apps) dans ce projet. Vous pourrez ensuite retrouver vos clés et jetons dans votre application développeur.
 

<div id="rate-limits">
  #### Limites de taux
</div>

Chaque jour, plusieurs milliers de développeurs envoient des requêtes à l’API X. Pour gérer le volume important de ces requêtes, des [limites de taux](/fr/x-api/fundamentals/rate-limits) sont appliquées à chaque endpoint, afin de restreindre le nombre de requêtes que vous pouvez effectuer au nom de votre application ou d’un utilisateur authentifié.

Les endpoints de consultation (GET) sont soumis à des limites de taux à la fois au niveau de l’application et de l’utilisateur, tandis que les endpoints de gestion (POST/DELETE) sont limités au niveau de l’utilisateur. La limite au niveau de l’application signifie que vous, en tant que développeur, ne pouvez effectuer qu’un certain nombre de requêtes vers cet endpoint sur une période donnée depuis n’importe quelle application (déterminée par l’utilisation de l’API Key et de l’API Secret Key, ou du App-only Access Token). La limite au niveau de l’utilisateur signifie que l’utilisateur authentifié pour lequel vous effectuez la requête ne peut effectuer une consultation de liste qu’un certain nombre de fois, quel que soit l’App du développeur.

Le tableau ci-dessous présente les limites de taux pour chaque endpoint.

|     |     |     |
| :--- | :--- | :--- |
| **Endpoint** | **Méthode HTTP** | **Limite de taux** |
| /2/lists/:id/members | GET | 900 requêtes par 15 minutes |
| /2/users/:id/list&#95;memberships | GET | 75 requêtes par 15 minutes |
| /2/lists/:id/members | POST | 300 requêtes par 15 minutes |
| /2/lists/:id/members/:user&#95;id | DELETE | 300 requêtes par 15 minutes |

<div id="fields-and-expansions">
  #### Champs et expansions
</div>

Le point de terminaison GET de l’API X v2 permet aux utilisateurs de sélectionner précisément les données à renvoyer par l’API à l’aide d’un ensemble d’outils appelés `fields` et `expansions`. Le paramètre `expansions` vous permet de développer les objets référencés dans la charge utile. Par exemple, consulter les membres d’une Liste permet de récupérer les [expansions](/fr/x-api/fundamentals/expansions) suivantes :

* `pinned_tweet_id`

Le paramètre `fields` vous permet de sélectionner exactement quels [champs](/fr/x-api/fundamentals/fields), au sein des différents objets de données, vous souhaitez recevoir. La consultation des membres d’une Liste renvoie principalement des objets utilisateur. Par défaut, l’objet utilisateur renvoie les champs id, name et username. Pour recevoir des champs supplémentaires tels que `user.created_at` ou `user.description`, vous devez les demander explicitement à l’aide du paramètre `user.fields`. 

Nous avons ajouté un guide sur l’utilisation des [champs et expansions](/fr/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

Le tableau ci-dessous présente les champs et expansions disponibles pour chaque point de terminaison de recherche :

|     |     |     |
| :--- | :--- | :--- |
| **Endpoint** | **Champs** | **Expansions** |
| /2/lists/:id/members | `user.fields`<br /><br />`tweet.fields` | `pinned_tweet_id` |
| /2/users/:id/list&#95;memberships | `list.fields`<br /><br />`user.fields` | `owner_id` |

<div id="pagination">
  #### Pagination
</div>

La recherche d’adhésions/membres peut renvoyer un grand volume de données. Pour garantir des résultats cohérents et performants à tout moment, nous utilisons la pagination. La pagination est une fonctionnalité des points de terminaison de X API v2 qui renvoient plus de résultats qu’une seule réponse ne peut en contenir. Dans ce cas, les données sont renvoyées sous forme de « pages ». Pour en savoir plus sur la façon de [parcourir les résultats par pages.](/fr/x-api/fundamentals/pagination)
