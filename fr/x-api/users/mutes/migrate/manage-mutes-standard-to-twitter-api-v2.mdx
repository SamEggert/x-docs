<div id="manage-mutes-standard-v11-compared-to-x-api-v2">
  ### Gérer les masquages : Standard v1.1 par rapport à X API v2
</div>

Si vous avez travaillé avec les points de terminaison standard v1.1 [POST mutes/users/create](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/post-mutes-users-create) et [POST mutes/users/destroy](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/post-mutes-users-destroy), l’objectif de ce guide est de vous aider à comprendre les similitudes et les différences entre les points de terminaison de gestion des masquages du standard v1.1 et de X API v2.

* **Similarités**
  * Contexte utilisateur OAuth 1.0a
* **Différences**
  * URL des points de terminaison
  * Exigences relatives à l’application et au projet
  * Méthodes HTTP
  * Paramètres de requête
     

<div id="similarities">
  #### Similarités
</div>

**Méthode d’authentification OAuth 1.0a en contexte utilisateur**

Les deux versions d’endpoint prennent en charge [OAuth 1.0a User Context](https://developer.x.com/content/developer-twitter/resources/fundamentals/authentication). Par conséquent, si vous utilisiez auparavant l’un des endpoints standard v1.1 pour gérer les sourdines, vous pouvez continuer à utiliser la même méthode d’authentification en migrant vers la version X API v2.

<div id="differences">
  #### Différences
</div>

**URL des endpoints**

* Endpoints Standard v1.1 :
  * POST https://api.x.com/1.1/mutes/users/create.json
    (mettre un utilisateur en sourdine)
  * POST https://api.x.com/1.1/mutes/users/destroy.json
    (retirer un utilisateur de la sourdine)
* Endpoint X API v2 :
  * POST https://api.x.com/2/users/:id/muting
    (mettre un utilisateur en sourdine)
  * DELETE https://api.x.com/2/users/:source_user_id/muting/:target_user_id
    (retirer un utilisateur de la sourdine) 

**Exigences relatives à l’App et au Project**

Les endpoints X API v2 exigent l’utilisation des identifiants d’une [developer App](/fr/resources/fundamentals/developer-apps) associée à un [Project](/fr/resources/fundamentals/projects) lors de l’authentification de vos requêtes. Tous les endpoints X API v1.1 peuvent utiliser des identifiants provenant d’Apps autonomes ou d’Apps associées à un Project.

**Paramètres de requête**

Les paramètres de requête Standard v1.1 suivants ont des équivalents dans X API v2 :

| Standard v1.1 | X API v2 |
| :--- | :--- |
| user_id | target_user_id |
| screen_name | Aucun équivalent |

Veuillez noter que les paramètres Standard v1.1 sont transmis en tant que paramètres de requête, tandis que les paramètres X API v2 sont transmis en tant que paramètres de corps (pour l’endpoint POST) ou paramètres de chemin (pour l’endpoint DELETE).

De plus, l’id de l’utilisateur qui met en sourdine un utilisateur cible n’est pas requis lors de l’utilisation des endpoints Standard v1.1, car les jetons d’accès transmis avec [OAuth 1.0a User Context](/fr/resources/fundamentals/authentication) permettaient de déterminer quel utilisateur initiait l’action de mise en sourdine/annulation de la sourdine.
