<div id="comparing-x-apis-filtered-stream-endpoints">
  ## Comparaison des endpoints de flux filtré de l’API X
</div>

Le groupe d’endpoints de flux filtré v2 remplace [l’endpoint standard v1.1 statuses/filter](https://developer.x.com/en/docs/x-api/v1/tweets/filter-realtime/api-reference/post-statuses-filter) et l’[API PowerTrack](/fr/x-api/enterprise-gnip-2.0/powertrack-api). Si vous avez du code, des applications ou des outils qui utilisent une ancienne version de l’endpoint de flux filtré et que vous envisagez de migrer vers le nouvel endpoint X API v2, cette comparaison peut vous aider à démarrer.

Consultez nos guides de migration plus détaillés :

[Migration de Standard v1.1 vers X API v2](fr/x-api/posts/filtered-stream#standard-v1-1-compared-to-x-api-v2)

[Migration de l’API PowerTrack vers X API v2](/fr/x-api/enterprise-gnip-2.0/powertrack-api#powertrack-operators)

Le tableau suivant compare les endpoints de diffusion filtrée proposés par X :
 

| **Description** | **Standard v1.1** | **PowerTrack API** | **X API v2** |
| :--- | :--- | :--- | :--- |
| Accès | X App | Nécessite un contrat et un compte Enterprise | Nécessite un compte développeur ([s’inscrire](https://developer.x.com/en/portal/petition/essential/basic-info)) et une [X App](/fr/resources/fundamentals/developer-apps) au sein d’un [projet](/fr/resources/fundamentals/projects) |
| :--- | :--- | :--- | :--- |
| Domaine hôte | ******https://stream.x.com****** | ******https://gnip-stream.x.com****** | ******https://api.x.com****** |
| Chemin d’endpoint | ******1.1/statuses/filter.json****** | ******/stream/powertrack/accounts/{gnip_account_name}/publishers/twitter/{stream_label}.json******<br /><br />******/rules/powertrack/accounts/{gnip_account_name}/publishers/twitter/{stream_label}.json ******<br /><br />******/rules/powertrack/accounts/{gnip_account_name}/publishers/twitter/{stream_label}/validation.json****** | ******/2/tweets/search/stream******<br /><br />******/2/tweets/search/stream/rules****** |
| [Authentification](/fr/resources/fundamentals/authentication) | OAuth 1.0a (contexte utilisateur) | Authentification HTTP Basic | OAuth 2.0 (application uniquement) |
| Méthodes HTTP prises en charge | POST | GET  <br />POST | GET  <br />POST |
| Paramètres requis | Règle définie dans la connexion en tant que paramètre, au moins un des éléments suivants :<br /><br />* ******follow******<br />* ******track******<br />* ******locations****** | Aucun paramètre requis pour la connexion en streaming, paramètre de backfill facultatif.<br /><br />Règles gérées séparément | Aucun paramètre requis pour la connexion en streaming, paramètres optionnels pour définir le format de réponse et ajouter la [fonction de récupération par backfill](/fr/x-api/posts/filtered-stream#recovering-missed-data-after-a-disconnection-backfill) pour l’accès Academic Research.<br /><br />Règles gérées séparément |

| Nombre maximal de connexions autorisées | 2 simultanées par utilisateur autorisé | Prend en charge des connexions multiples/redondantes, déterminé par contrat | Accès Pro :  <br />1 |
| [Fonctionnalités de récupération et de redondance](/fr/x-api/posts/filtered-stream#filtered-stream-recovery-and-redundancy-features) | Aucune | Backfill, connexions redondantes et API Replay |     |
| [Plafonds de posts](/fr/x-api/fundamentals/post-cap) | Limité à 1 % du firehose | Déterminé par contrat | Un plafond mensuel de posts au niveau du projet est appliqué à tous les posts reçus depuis cet endpoint :<br /><br />Basic :  <br />10 000 posts<br /><br />Pro :  <br />1 million de posts |
| Signal de maintien en vie/heartbeats | lignes vides (\r\n ou similaire) au moins toutes les 20 secondes | lignes vides (\r\n ou similaire) toutes les 10 secondes | lignes vides (\r\n ou similaire) au moins toutes les 20 secondes |
| Latence | 10 secondes | 2 secondes<br /><br />Au moins 10 secondes pour le développement d’URL (URL unwinding) | 10 secondes |
