<div id="typical-response-structure">
  ## Structure de réponse typique
</div>

Les réponses réussies sont indiquées par un code HTTP de la série 200 et une charge utile JSON contenant l’objet ou les objets demandés, créés, modifiés ou supprimés, ainsi qu’une représentation de l’interprétation de votre requête par le serveur.

Si votre requête aboutit, votre réponse inclura un nœud `request` répliquant votre requête.

*Exemple :* [GET accounts/abcdefg/campaigns?with_deleted=true](/fr/x-ads-api/campaign-management#campaigns)

```json
{
  /* the data of your response... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

Le champ `data` des réponses JSON contient les objets spécifiques associés à la ressource sollicitée. Le nœud `data` est un tableau JSON lorsque la réponse peut contenir un ou plusieurs résultats. Il est renvoyé sous forme de dictionnaire (hash) JSON lorsqu’un seul résultat est possible. Dans de rares cas, une réponse qui inclurait normalement une collection peut contenir à la place une seule hashmap. Dans ce cas, supposez que cette hashmap unique est un objet du même type que celui indiqué dans le champ `type`.

<div id="error-response-structure">
  ## Structure de la réponse d’erreur
</div>

Les réponses d’erreur sont renvoyées avec un code HTTP hors de la série 200. En général, une réponse JSON est jointe, mais certaines erreurs renvoient un corps d’un autre type. Dans les cas où la structure de la réponse ne peut pas être analysée, considérez que la signification fondamentale du code HTTP prime. Par exemple, vous pouvez occasionnellement voir un HTTP 404 accompagné d’une réponse HTML. Dans ce cas, il est raisonnable de supposer que le contenu est introuvable (HTTP 404 signifie « Not Found »).

Les réponses d’erreur typiques suivent une structure similaire aux réponses réussies. La nature de l’erreur est communiquée dans un nœud `errors` de la réponse. Le nœud `errors/code` indique un code d’erreur constant en CAPS_CASE que vous pouvez traiter de manière programmatique pour prendre des décisions de résolution. Le nœud `errors/message` indique une description (généralement) lisible par l’humain de l’erreur en anglais. Des champs supplémentaires peuvent être ajoutés pour fournir des détails plus précis sur l’erreur.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

Dans l’exemple ci-dessus, une requête vers un point de terminaison d’analytique a été effectuée avec une valeur non valide pour le paramètre `start_time`. Le `errors/code` pour les requêtes avec des paramètres non valides est `INVALID_PARAMETER`.
