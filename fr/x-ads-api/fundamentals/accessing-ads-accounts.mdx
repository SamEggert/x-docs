<div id="user-accounts-vs-ad-accounts">
  # Comptes utilisateurs vs comptes publicitaires
</div>

Deux types de comptes interviennent dans l’utilisation de l’Ads API : les comptes publicitaires et les comptes utilisateurs X. Dans toute la documentation de l’Ads API, le terme « compte » fait généralement référence au compte publicitaire.

* Les comptes publicitaires sont enregistrés sur **business.x.com** et identifiés dans l’API par account&#95;id. Ils sont directement liés à des sources de financement et s’appuient sur le contenu d’un ou plusieurs comptes utilisateurs X en tant qu’« utilisateurs promouvables ». Chaque compte publicitaire peut accorder des autorisations à un ou plusieurs comptes utilisateurs X. Le compte publicitaire, ou « compte actuel », est présent dans quasiment chaque URL exécutée sous la forme d’un paramètre en ligne :account&#95;id.
* Les comptes utilisateurs X (tels que @AdsAPI) sont identifiés par user&#95;id dans l’Ads API. Un ou plusieurs de ces comptes peuvent être associés à un compte publicitaire. Le compte utilisateur X authentifié qui effectue des requêtes sur l’API est appelé « utilisateur actuel ». La liste des comptes publicitaires auxquels l’utilisateur actuel a accès est disponible via [GET accounts](/fr/x-ads-api/campaign-management#accounts). Les « utilisateurs promouvables » sont des handles X que peut promouvoir un compte publicitaire spécifique. Pour plus de détails, consultez [Obtaining Ads Account Access](/fr/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Méthodes d’accès au compte publicitaire
</div>

Il existe deux méthodes pour effectuer des requêtes Ads API pour le compte d’un annonceur :

1. Effectuer des requêtes au nom d’un annonceur (**recommandé**)
2. Effectuer des requêtes en utilisant votre compte auquel l’accès au compte de l’annonceur a été accordé, par exemple une [agence](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) gérant plusieurs comptes.

Ce document propose un bref aperçu des différences entre ces options et doit être utilisé conjointement avec nos autres ressources, telles que la [FAQ sur la connexion multi-utilisateur](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html).

Comme décrit dans [Autoriser une requête](/fr/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), toutes les requêtes vers l’Ads API nécessitent des en-têtes d’autorisation utilisant [OAuth 1.0a](/fr/resources/fundamentals/authentication) avec un jeton d’accès obtenu via le flux [OAuth à 3 intervenants](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) . Votre application devra implémenter un flux OAuth web pour [obtenir des jetons d’accès](/fr/resources/fundamentals/authentication#oauth-1-0a-2). Les développeurs Ads API ne doivent jamais demander à nos annonceurs X de partager leurs identifiants de connexion.

Par défaut, chaque [application développeur X](/fr/resources/fundamentals/developer-apps) contient un **jeton d’accès** statique qui peut être utilisé pour effectuer des requêtes Ads API pour le compte propriétaire de l’application. Ces identifiants sont idéaux pour des cas d’usage mono-compte, sans nécessiter un flux OAuth à 3 intervenants ou [basé sur un code PIN](/fr/resources/fundamentals/authentication#pin-based-authorization). Si vous n’accédez pas à un autre compte X Ads, utilisez ces identifiants [monoutilisateur](/fr/resources/fundamentals/authentication#oauth-1-0a-2) au lieu des étapes suivantes.

<div id="levels-of-access">
  ## Niveaux d’accès
</div>

<div id="app-level-permissions">
  ### Autorisations au niveau de l’application
</div>

Chaque utilisateur disposera d’un niveau d’accès tel que défini lors de la [demande d’accès à l’Ads API](/fr/x-ads-api/getting-started/step-by-step-guide) :

* **Conversions uniquement** : l’utilisateur a accès aux points de terminaison [Mobile](/fr/x-ads-api/measurement/mobile-conversions) et [Web Conversion](/fr/x-ads-api/measurement/web-conversions) avec des autorisations de [lecture et écriture](/fr/resources/fundamentals/developer-apps).
* **Accès standard** : l’utilisateur a accès aux points de terminaison [Analytics](/fr/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) et [Conversion](/fr/x-ads-api/measurement/web-conversions) avec des autorisations de [lecture et écriture](/fr/resources/fundamentals/developer-apps).

**Remarque :** Les développeurs Ads API ayant demandé l’accès avant juillet 2023 peuvent disposer de niveaux d’accès et d’autorisations différents et peuvent être limités à cinq jetons OAuth. Consultez notre guide sur [l’augmentation des accès](/fr/x-ads-api/getting-started/increasing-access) pour obtenir l’accès à des points de terminaison supplémentaires ou lever les limites de jetons pour les applications existantes.

<div id="ad-account-level-permissions">
  ### Autorisations au niveau du compte publicitaire
</div>

Chaque utilisateur ayant accès à un compte publicitaire possède une autorisation spécifique au niveau du compte : **Account administrator**, **Ad manager**, **Campaign analyst**, **Organic analyst** et **Creative Manager** ; consultez **business.x.com** pour la documentation la plus récente sur les autorisations au niveau du compte. Les applications doivent récupérer les autorisations de l’utilisateur actuellement authentifié via le point de terminaison d’API [Authenticated User Access](/fr/x-ads-api/campaign-management#authenticated-user-access) afin de déterminer quels points de terminaison d’API et quelles fonctionnalités publicitaires sont accessibles.

**Remarque :** Tout jeton utilisateur utilisé avec la [Conversion API](/fr/x-ads-api/measurement/web-conversions) doit appartenir à des utilisateurs disposant des autorisations au niveau du compte **Account administrator** ou **Ad manager**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Méthodes pour obtenir des jetons d’accès
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtenir le jeton d’accès d’un annonceur (utilisateur)
</div>

Il existe deux méthodes pour obtenir un **jeton d’accès** d’annonceur. La plus courante consiste à utiliser un [flux OAuth à 3 étapes](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) directement depuis votre interface Web. Les applications qui n’ont pas d’interface publique accessible aux annonceurs peuvent implémenter un processus OAuth [basé sur un code PIN](/fr/resources/fundamentals/authentication#pin-based-authorization). Une fois que l’utilisateur a terminé le flux à 3 étapes, votre application disposera des identifiants nécessaires pour effectuer des requêtes sur son compte Ads via l’API.

L’obtention des identifiants utilisateur via un flux OAuth est la méthode que nous **recommandons vivement** à la majorité des développeurs de l’Ads API pour accéder à un compte annonceur. Elle vous permet d’appeler l’API au nom d’un utilisateur et d’agir en tant que cet utilisateur. Ces jetons n’expirent pas, mais peuvent être révoqués par l’utilisateur à tout moment.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtenez votre jeton d’accès (développeur)
</div>

Cette option nécessite que l’annonceur [accorde à votre @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (ou à vos @usernames) l’accès à son compte X Ads via l’interface X sur business.x.com. Les **jetons d’accès** obtenus via le flux OAuth à 3 volets pour votre compte pourront accéder au compte X Ads de l’annonceur.

Cela vous permet d’appeler l’API en utilisant les jetons OAuth de votre propre @username plutôt que ceux de l’annonceur. La distinction clé avec cette option est que vous ne pouvez créer des Promoted-Only Posts que si l’autorisation de délégation/composition de Post a été accordée à votre @username.

Pour obtenir les droits de création de Promoted-Only Posts au nom de l’utilisateur promouvable `FULL` du compte, vous devez également être autorisé à créer des Posts dans ce flux. Cela activera l’accès via l’autorisation `TWEET_COMPOSER` sur l’endpoint [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](/fr/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Différences entre ces méthodes
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Jeton OAuth d’annonceur (utilisateur) | Jeton OAuth (développeur)  <br /> (@username ajouté pour séparer le compte) |
| Accès au compte Ads | ✔   | ✔   |
| Créer des posts au nom de l’utilisateur | ✔   | ✔*  |
| Gérer les campagnes | ✔   | ✔   |
| Accéder aux analytics | ✔   | ✔   |
| Créer des cards au nom de l’utilisateur | ✔   | ✔   |
| Possibilité pour le développeur d’accéder  <br />via l’interface X Ads |     | ✔   |
| [Limites de taux](/fr/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Distinctes par annonceur | Distinctes par compte annonceur |
